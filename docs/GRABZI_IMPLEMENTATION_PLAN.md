# GRABZI — план реализации (v7, исполняемый)

> **Версия v7:** проект = **готовый бэкенд+админка Juicy, доработанные под GRABZI, + отдельный фронт GRABZI** (§A). v6: видимость после оплаты на `payment_status`, авто-логин по телефону без OTP. **v7 закрывает пробелы админки** (аудит Juicy): загрузка медиа, CMS контента, операционная пауза точки, остаток на экране менеджера, граница «админка vs конфиг» — §5.11.

> **Суть (главное).** Делаем проект **GRABZI на базе готового кода Juicy**: берём **бэкенд и админку Juicy** как основу и **дорабатываем под требования GRABZI** (локации, лимиты, часы, привязка сотрудников, аналитика по точкам). Поверх — **отдельный фронтенд GRABZI** в дизайне GRABZI (desktop + mobile), подключённый к этому бэкенду. Что в GRABZI не нужно (добавки, купоны, онбординг, ввод имени, корзина) — просто не используем; код Juicy не ломаем.
>
> **Источники.** Парсинг `grabzi.ae` (Shopify) и `order.grabzi.ae` (React/Supabase/Stripe SPA), аудит проекта Juicy (`app/` — Next.js app-router, `backend/` — FastAPI + SQLAlchemy). Данные парсинга — в `grabzi_parser/`.
>
> Документ самодостаточен: подход, модель данных, контракты API, карта экранов и компонентов, фазы с критериями приёмки.

| # | Решение |
|---|---|

| Р2 | **Две версии вёрстки — desktop и mobile** (адаптив/респонсив, см. §3). |
| Р0 | **Берём готовый бэкенд+админку Juicy как основу и дорабатываем под GRABZI.** Не пишем заново — используем существующий код Juicy (FastAPI+SQLAlchemy + админка) и расширяем его требованиями GRABZI (локации, лимиты, часы, привязка сотрудников, аналитика). | Р3 | **Без добавок.** Публичный каталог = категории напитков + напитки (тот же формат карточек, видео-петли). Добавки/KБЖУ-добавки **не выводим**. Логика **каталога берётся из Juicy**. Бэкенд-сущности добавок дремлющие (§4). |

| Р4 | **Локации — first-class.** Админка создаёт локации (описание, адрес, часы по дням, лимит); публично — выбор локации с остатком перед созданием заказа; при оплате — атомарная проверка лимита. |
| Р5 | **Заказ виден менеджеру и списывает лимит ТОЛЬКО после оплаты.** На «оплатить» создаётся заказ с `payment_status="pending"` + платёж; **до оплаты он скрыт от менеджеров и из статистики** (Juicy уже фильтрует по `payment_status="paid"`), **лимит не трогает**. После успешной оплаты → `payment_status="paid"` (доступен к приготовлению) + списывает лимит. Это исключает «мусорные» неоплаченные заказы. Лимит считается **в напитках** (`sum quantity`); refund **возвращает** напитки в лимит того дня (§5.4). **Заказ фиксирует `location_id`** (аналитика/история). Реализуется на существующем `payment_status` Juicy — без нового статуса (§5.1). |
| Р5.1 | **Дневной лимит — необязательный.** Задан → проверяется; **не задан (`null`) → без лимита** (объём всё равно учитывается для аналитики). |
| Р6 | **Рабочие часы локации блокируют заказ.** Вне окна работы точки заказ недоступен (per-location `is_open`). |
| Р7 | **Каталог одинаков для всех локаций** (матрицу «напиток × локация» не делаем). |
| Р8 | **Купоны — НЕ нужны** в GRABZI. Оценка 👍/👎 **остаётся**, но **без выпуска купона** (в Juicy дизлайк выпускал купон — для GRABZI отключаем). Купонные сущности дремлющие. |
| Р9 | **Онбординг и «ввод имени после регистрации» — НЕ нужны** (ни Juicy, ни GRABZI). Убираем из флоу. |
| Р10 | **«Мои заказы»** с двумя состояниями: авторизован → список заказов; не авторизован → предложение создать заказ **или войти по номеру телефона** (тот же номер, что в прошлых заказах) для подгрузки истории (§4.4). |
| Р11 | **Профиль / личный кабинет** — как в основном Juicy: имя, авто, настройки. **Без купонов.** |
| Р12 | **Публичный сайт GRABZI — только английский.** Переключатель скрыт. Админка — RU/AR временно. Арабская версия — отдельный отложенный модуль (§3.3), сейчас только дешёвый no-regret задел. |
| Р13 | **Сотрудник привязывается к локации.** Супер-админ прикрепляет сотрудника к точке. **Менеджер видит/обрабатывает заказы только своей локации**; супер-админ — все. Локация менеджера видна в аналитике (§5.9). |

**Открытые (не блокируют старт, есть дефолты):**
- Имя инфо-страницы: дефолт `/info`. Контент из админки: дефолт — да (`info_blocks`).
- Деталка напитка (Р3.2): дефолт — модуль **выключен** на старте, включается флагом.

## A. Подход: дорабатываем готовый бэкенд Juicy под GRABZI (Р0)

**Логика простая, как у разработчика:**
- Берём **готовый бэкенд и админку Juicy** (FastAPI+SQLAlchemy + админ-панель) — это **основа** проекта GRABZI.
- **Дорабатываем их под требования GRABZI:** добавляем локации, дневные лимиты, рабочие часы по дням, привязку сотрудников к точке, доступ менеджера к своей точке, аналитику по локациям (всё ниже по документу).
- Делаем **отдельный фронтенд GRABZI** в дизайне GRABZI и подключаем к этому бэкенду.
- То, что у GRABZI **не нужно** (добавки, купоны, онбординг, ввод имени, корзина) — **просто не используем** на фронте GRABZI; в коде Juicy это остаётся нетронутым (ничего не удаляем, чтобы не ломать).

**Что меняем, что не трогаем:**
- *Меняем/добавляем (бэкенд+админка):* новый раздел «Локации», поля и счётчики лимитов, часы по дням, `orders.location_id` + списание лимита в `mark_paid` (на существующем `payment_status`), гостевой заказ по телефону (§4.7), привязка сотрудника к локации, фильтр заказов по локации для менеджера, аналитика по точкам.
- *Не трогаем:* модели и эндпоинты каталога/заказов/оплат/пользователей — переиспользуем как есть.
- *Фронт GRABZI:* новый, берёт логику экранов из Juicy, но рисует на компонентах GRABZI и обращается к доработанному бэкенду.

> Без env-переключателей брендов и хитрых схем развёртывания: это **проект GRABZI на базе кода Juicy**. Juicy-проект живёт своей жизнью отдельно; здесь мы делаем GRABZI, переиспользуя его готовый бэкенд.

## 2. Стратегия сборки: что воспроизводим, что добавляем

Ключевой принцип (Р1): **два класса экранов.**

- **Класс A — «есть в GRABZI» → воспроизводим как есть** на нашем стеке: верстаем по спарсенному дизайну GRABZI (бренд, шрифты, цвета, контент), подключаем к нашему API. Не редизайним.
- **Класс B — «нет в GRABZI, есть в Juicy» → добавляем недостающее**: берём из Juicy **логику и структуру экрана**, но **перебираем на компонентах GRABZI** (UI-кит GRABZI), в desktop+mobile.

Колонка «Действие» в §3 размечена этими классами.

## 3. Платформа и дизайн-система

### 3.1. Desktop + mobile (Р2)
- **Подход:** единый адаптивный фронт (Next.js), брейкпоинты `mobile < 768 ≤ tablet < 1024 ≤ desktop`. Не делаем отдельных кодовых баз.
- **Навигация:**
  - mobile — нижний таб-бар (GRABZI-аналог `BottomNav`): Home / Menu / Orders / Profile;
  - desktop — верхний header-nav (новый компонент `GrabziHeader`) с теми же разделами + CTA «Create order» и выбранная локация с остатком (корзины нет — заказ одностраничный, Р3.1).
- **Сетки:** каталог — 2 колонки на mobile, 3–4 на desktop; деталка напитка — одна колонка на mobile, две (медиа слева / инфо справа) на desktop.
- **Тач/ховер:** на desktop добавляем ховер-состояния карточек и видео-превью по hover; на mobile — автоплей-петля в карточке как у GRABZI.

### 3.2. UI-кит GRABZI (Р1) и карта компонентов
Заводим брендовый кит `app/components/grabzi/*` (или тему `data-brand="grabzi"` + `app/styles/brand-grabzi.css`). Токены:
- Шрифты: `Grenadine MVB Black` (900) и `Grenadine MVB Extra Bold Italic` (заголовки/акценты).
- Акцент: `#c44429` (терракота, из CTA/`📍 LOCATION`).
- Тон: эмодзи-нейминг напитков (`🍒💣 🧊 🍫🍇 🍉⚡️`), drive-through-настроение, EN-only.

**Соответствие компонентов (Juicy → GRABZI-кит):**
| Назначение | Juicy-компонент | GRABZI-замена |
|---|---|---|
| Шапка экрана | `TopBar` | `GrabziTopBar` (mobile) / `GrabziHeader` (desktop) |
| Нижняя навигация | `BottomNav` | `GrabziBottomNav` (mobile) + header (desktop) |
| Карточка напитка | `ApiProductCard` | `GrabziDrinkCard` (медиа-петля, цена, эмодзи-имя) |
| Степпер кол-ва | `StepperButton` | `GrabziStepper` |
| Модалка/шторка | `BottomSheet` / `Modal` | `GrabziSheet` / `GrabziModal` |
| Тосты | `useToast` | переиспользуем, ре-стайл |

**Админка** компонентов GRABZI **не получает** — остаётся на `AdminShell`/`Modal`/`admin-*` стилях Juicy (Р8).

### 3.3. Арабская локализация — отдельный отложенный модуль (Р9)
**Статус: под вопросом у заказчика, в текущий скоуп НЕ входит.** База публичного сайта — **чисто английская**. Чтобы будущий арабский не потребовал переписывания, делаем **только дешёвый no-regret задел** (он почти не удорожает EN-разработку), а всё «дорогое» по AR выносим в отдельный модуль, который запускается **только по подтверждению заказчика**.

**A. Обязательный сейчас (дешёвый задел, без удорожания):**
- **Никакого хардкода UI-строк** в JSX — все строки через словарь `messages/en.json` (структура ключей готова, заполнен только `en`).
- **Логические CSS-свойства** вместо физических (`margin-inline-start/end`, `padding-inline`, `text-align: start/end`, `inset-inline`) — это и для LTR корректно, и даёт RTL «бесплатно».
- **Контент-поля в БД — JSON i18n** `{ "en": "..." }` (схема Juicy уже такая) — добавление `ar` позже не меняет схему.
- API контента принимает `?locale=` с фолбэком на `en`.

**B. Отдельный модуль «Арабская версия» (НЕ сейчас, только если заказчик подтвердит):**
- Локаль-роутинг `/(en|ar)/`, флаг `public_locales=["en","ar"]`, `<html lang dir>`.
- RTL-вёрстка и приёмка всех публичных экранов в `dir="rtl"`, зеркалирование иконок/прогресса.
- Заполнение `messages/ar.json` и `ar`-контента (переводы напитков/инфо).
- Арабский шрифтовой стек (напр. `IBM Plex Sans Arabic`/`Noto Kufi Arabic`), отдельные типографские токены, условное подключение по `lang="ar"`.
- Переключатель языка в публичном UI, форматирование чисел/дат через `Intl` для `ar`.

> Граница ответственности: пункт **A делаем в Фазе 1** (почти бесплатно), пункт **B — отдельная оценка и фаза, вне базового скоупа**. Так EN→EN+AR останется «наполнение данными + флаг + RTL-приёмка», но мы не платим за это сейчас.

**Маршрутизация и определение языка**
- Закладываем локаль-сегмент в URL: `/(en|ar)/...` (Next.js i18n routing / middleware). Сейчас активна только `en`; `ar` зарезервирована и скрыта за флагом `public_locales=["en"]` (на будущее `["en","ar"]`).
- Источник локали: сегмент URL → `<html lang>` и `dir`. EN → `dir="ltr"`, AR → `dir="rtl"`.

**RTL-вёрстка (главное для AR)**
- **Никаких физических `left/right/margin-left/padding-right`** в стилях публичной части — только **логические свойства** (`margin-inline-start/end`, `inset-inline`, `text-align: start/end`, `padding-inline`). Это даёт зеркалирование «бесплатно» при `dir="rtl"`.
- Иконки и направленные элементы (стрелки «назад», прогресс заказа, карусели) — зеркалируются по `:dir(rtl)` / `[dir="rtl"]`-селекторам.
- Flex/grid — направление потока через `flex-direction`/логические выравнивания, чтобы при RTL не ломалось.
- Тестовый чек: каждый публичный экран проверяется в `dir="rtl"` (storybook-knob / тестовый флаг) уже сейчас, чтобы не накопить LTR-долг.

**Тексты UI**
- Все строки интерфейса — **через словарь переводов** (`messages/en.json`, зарезервирован `messages/ar.json`), **без хардкода** строк в JSX. Сейчас заполнен только `en`. Используем `next-intl` / лёгкий собственный словарь.
- Числа/даты/валюта — через `Intl` с локалью (AED форматирование корректно и для `ar`).

**Контент (напитки, инфо-страница)**
- Поля контента в БД делаем **локализуемыми**: `drinks.name`, `drinks.description`, `drink_categories.name`, `info_blocks.title/body` хранятся как JSON `{ "en": "...", "ar": "..." }` (инфраструктура i18n Juicy уже такая). Сейчас заполняем только `en`; `ar` добавляется позже без миграции схемы.
- API отдаёт контент по запрошенной локали (`?locale=en|ar`), с фолбэком на `en`.

**Шрифты**
- Латинский бренд-шрифт `Grenadine MVB` не покрывает арабицу. Заранее закладываем **арабский шрифтовой стек** (напр. `IBM Plex Sans Arabic` / `Noto Kufi Arabic`) и подключаем его условно при `lang="ar"`. Размеры/межстрочные — отдельные токены для AR (арабская типографика «выше»).

**Что делаем сейчас vs потом**
- *Сейчас:* словарь `en`, логические CSS-свойства, локаль-роутинг с одной активной локалью, JSON-контент с ключом `en`, RTL-прогон в тестах.
- *Потом (включение AR):* заполнить `messages/ar.json` и `ar`-контент, добавить `ar` в `public_locales`, подключить арабский шрифт, включить переключатель языка, прогнать RTL-приёмку.

> Итог: переход EN → EN+AR — это **наполнение данными и включение флага**, без рефакторинга вёрстки и схемы.

## 4. Публичный флоу GRABZI: каталог, создание заказа, мои заказы

Логику **каталога берём из Juicy**, но публичный UX упрощаем под GRABZI: без корзины, без обязательной деталки, без добавок и купонов.

### 4.1. Главная
- Hero/бренд GRABZI + кнопка **«Create order»**. Клик → **сначала выбор локации** (§5.7), затем страница создания заказа.

### 4.2. Создание заказа — одна страница (Р3.1)
Единый экран (аналог `order.grabzi.ae`, но мультилокационный):
- Сверху — выбранная локация + индикатор **остатка** («N left today» или «No limit») и статус часов.
- **Табы категорий напитков**; при выборе категории **сразу виден список всех её напитков** на этой же странице (медиа-петля, имя с эмодзи, цена), у каждого **`+/–` счётчик количества**.
- Снизу — сводка: позиции, сумма; поля **телефон** и **номер авто** (предзаполняются из профиля/прошлых заказов, если есть); кнопка **Pay**.
- **Контроль лимита в реальном времени:** если суммарное число напитков в заказе превышает текущий `remaining` локации — показываем **модалку/сообщение** «Only N drinks left at this location» и блокируем оплату, пока пользователь не уменьшит количество (или не сменит локацию). Остаток обновляется (WS/поллинг §5.7).
- Без добавок; цена позиции = `base_price × qty`.

### 4.3. Деталка напитка — опциональный модуль (Р3.2)
Не обязателен (флаг `product_detail_enabled`, по умолчанию off). Если включён — отдельный экран `/product/[slug]` (как в основном проекте): медиа, описание, KБЖУ самого напитка, счётчик количества и кнопка **«Create order»**, которая ведёт в §4.2 с предзаполнением (этот напиток + qty, телефон, авто) и показом доступного лимита.

### 4.4. Мои заказы (Р10)
Экран `/orders`:
- **Авторизован** → список его заказов (номер, состав, статус, сумма, оценка).
- **Не авторизован** → сообщение «Create your first order or sign in», и форма **ввода номера телефона** → авто-логин по телефону без OTP (§4.7) → показываем историю заказов этого пользователя.

### 4.5. Профиль (Р11)
Личный кабинет как в основном Juicy: **имя**, авто/эмират, настройки. **Без купонов**, без переключателя языка (EN-only).

### 4.6. Бэкенд каталога/заказа (переиспользование Juicy, минимум изменений)
- Каталог: модели/эндпоинты `categories`, `drinks` Juicy — **как есть**.
- Добавки (`Addon/AddonCategory/DrinkAddon/OrderItemAddon`) и купоны (`Coupon`) — **дремлющие**: в GRABZI-флоу не используются (флаги `catalog_addons_enabled=false`, `coupons_enabled=false`). Заказ создаётся без addon-строк и без выпуска купонов. Никаких удалений схемы — чистое переиспользование.
- `POST /drinks/{slug}/preview` на GRABZI не вызывается; цена = `base_price × qty`.
- **Оценка** (`rate`) остаётся, но при `coupons_enabled=false` дизлайк **не выпускает купон** (§Р8).

**Админ-каталог (админка Juicy, для GRABZI):**
- Активны **Categories** и **Products**.
- **Addons/Groups/Units** и **Coupons** — в навигации GRABZI не показываем (код Juicy остаётся, маршруты живут).
- В редакторе напитка скрыт блок «привязка добавок».

### 4.7. Авторизация по телефону без OTP (авто-логин)
**Факт по коду Juicy:** `POST /api/orders` **требует авторизации** (`Depends(get_current_user)`), а `Order.user_id` — **NOT NULL**. Сохраняем это требование, но делаем вход **бесшовным по номеру телефона, без OTP и без пароля** (быстрый drive-through). OTP сейчас **не нужен** (остаётся в коде Juicy за выключенным флагом `auth_otp_enabled=false`).

**Как работает (переиспользуем `auth/verify` Juicy, выключив проверку кода):**
- На странице заказа (§4.2) пользователь вводит **телефон + авто**. При «Pay»:
  - **телефон новый** → создаём пользователя по этому номеру (без пароля/кода), сразу **выдаём токен** (авто-логин) и привязываем заказ к нему;
  - **телефон уже был** → находим существующего пользователя с этим номером, **авторизуем его** (выдаём токен) и привязываем новый заказ к нему — так все заказы с этим телефоном собираются у одного юзера.
- `user_id` всегда проставлен (NOT NULL соблюдён), схема Juicy не меняется. По сути это `auth/verify` Juicy в режиме «без кода»: find-or-create по телефону + токен.

**«Мои заказы» (§4.4):**
- Не авторизован → вводит свой номер телефона → тем же механизмом (find-or-create + токен) **авторизуется и видит свою историю** заказов по этому номеру. Без OTP.

> **Известный компромисс (сейчас осознанно принят):** без OTP вход по одному номеру не подтверждает владение им — теоретически можно ввести чужой номер и увидеть его историю. Для текущего drive-through-MVP это допустимо. **Усиление на будущее:** включить `auth_otp_enabled=true` (код в Juicy уже есть) — тогда вход/история потребуют подтверждение номера кодом. Зафиксировано в §12.

- **новый статус НЕ вводим.** В Juicy у заказа уже есть два независимых поля: `status` (`new|in_progress|ready|completed|refund` — стадия готовки) и `payment_status` (`pending|paid|failed|refunded`). Менеджерский список (`admin_orders`) **уже фильтрует `payment_status == "paid"`** — то есть неоплаченный заказ (`payment_status="pending"`) **и так невидим менеджеру и в статистике**. Поэтому «видимость только после оплаты» (Р5) достигается **существующим `payment_status`**, без новой сущности и нового значения статуса. - зависшие неоплаченные (`payment_status="pending"` старше TTL) — фоновая чистка в `payment_status="failed"` (значение уже есть); они и так скрыты, чистка лишь убирает «висяки».

> Это **максимальное переиспользование**: модель Juicy уже разделяет готовку и оплату, а менеджерский фильтр по `paid` даёт гарантию «нет мусорных заказов» из коробки. Нам остаётся добавить только `location_id` и списание лимита в `mark_paid`.

**`info_blocks`** (для §6) — `key`, `title` (JSON i18n), `body` (JSON i18n, richtext), `sort`, `is_active`.

### 5.2. Логика остатка (с учётом «без лимита», Р5.1)
```
sold_today(loc)   = counter[loc, today(loc.tz)].committed_drinks   (0 если строки нет)
если loc.daily_drink_limit == null:        # лимита нет
    remaining(loc)   = null (∞);  is_sold_out(loc) = false
иначе:
    remaining(loc)   = max(0, loc.daily_drink_limit - sold_today(loc))
    is_sold_out(loc) = remaining(loc) == 0
```
Граница суток — **в TZ локации**, не в UTC. **`committed_drinks` инкрементируется всегда** (даже при `null`-лимите) — чтобы супер-админ видел объём по точке в аналитике (§5.9). Проверка лимита при оплате (§5.3) **пропускается**, если лимит `null`.

### 5.3. Поток оплаты: заказ виден менеджеру и списывает лимит только после оплаты (Р5)
Используем **существующий `payment_status`** Juicy (см. §5.1): неоплаченный заказ (`pending`) уже невидим менеджеру (фильтр `paid`) и не в статистике; лимит списываем в момент оплаты.

Поток (как в Juicy + `location_id` + списание лимита):
1. **«Pay»** — `POST /api/orders` (тело Juicy + `location_id`). Сервер:
   - **мягкая предпроверка**: точка существует, `is_open`, и (если лимит задан) `qty ≤ remaining`; иначе `409` (`LOCATION_CLOSED` / `LOCATION_LIMIT_EXCEEDED` / `LOCATION_SOLD_OUT`) — до создания платежа;
   - создаёт заказ `status="new"`, `payment_status="pending"` (snapshot позиций/цен, `location_id`), **лимит НЕ списывает** (заказ и так скрыт фильтром `paid`).
2. **`POST /api/payments/checkout-session`** (как в Juicy) — Stripe-сессия с `order_id` в `metadata`; возвращает checkout URL (или mock-подтверждение в dev). Оплата — Stripe/Apple Pay.
3. **Вебхук `checkout.session.completed`** (`mark_paid(order_id)`) — **в одной транзакции**:
   1. `SELECT ... FOR UPDATE` строки `counter[location_id, today(tz)]` (создать с 0) — блок строки;
   2. перепроверка `is_open`; и, **если лимит задан**, `committed_drinks + drinks ≤ daily_drink_limit` (при `null` — пропуск);
   3. **успех** → `payment(succeeded)`, **`payment_status="paid"`** (теперь заказ виден менеджеру и в статистике), `committed_drinks += drinks`, события `paid`; коммит;
   4. **провал лимита/часов** (редкая гонка на последние слоты) → `payment_status="refunded"` (остаётся скрытым), **возврат Stripe**, лог `limit_exceeded_refund`; пользователю — «Sold out / closed, refunded».
4. **Idempotency:** по `order_id`/`session.id` повторный вебхук не списывает лимит дважды.
5. **Уборка зависших** (`payment_status="pending"` старше TTL) — фон в `payment_status="failed"` (значение уже есть в Juicy).

> Гарантия: лимит уменьшается и заказ появляется у менеджера **строго после получения денег** — на существующем механизме `payment_status` Juicy, минимум нового кода. До оплаты `remaining` на экране — справочный (без резерва).

### 5.4. Возвраты и лимит (Р5)
При `refund` оплаченного заказа: под тем же row-lock `committed_drinks -= drinks` для дня, **если** возврат происходит в тот же локальный день, что и продажа (иначе лимит прошлого дня не трогаем — он уже «сгорел»). Событие `refund` уже есть в Juicy; добавляем декремент счётчика.

### 5.5. Время открытия/закрытия по дням недели → блокировка заказа (Р6)
Супер-админ задаёт для каждой локации **время открытия и закрытия отдельно по каждому дню недели** (или помечает день выходным). Это (а) влияет на возможность создать заказ в конкретный момент и (б) выводится на публичной странице списка локаций.

- **Формат `working_hours`** — объект по 7 дням, у каждого дня `open`/`close` (или несколько интервалов, или `closed`):
  ```json
  {
    "mon": [{"open":"05:30","close":"22:00"}],
    "tue": [{"open":"05:30","close":"22:00"}],
    "wed": [{"open":"05:30","close":"22:00"}],
    "thu": [{"open":"05:30","close":"22:00"}],
    "fri": [{"open":"05:30","close":"22:00"}],
    "sat": [{"open":"05:30","close":"22:00"}],
    "sun": [{"open":"10:00","close":"18:00"}]
  }
  ```
  Массив интервалов поддерживает перерывы в течение дня; пустой массив `[]` = выходной. Дефолт-сид — из спарсенных часов GRABZI.
- **Редактор в админке (§5.8):** строка на каждый день недели с полями «открытие»/«закрытие» и чекбоксом «выходной»; кнопка «применить ко всем дням».
- `schedule_open(loc, now)` и `next_open_at` — вычисляются в `loc.timezone`.
- **Итоговая доступность точки:** `is_open = is_active && accepting_orders && schedule_open(now)` (учитывает и расписание, и ручную паузу §5.11.3).
- **Публичный справочный API** `GET /api/locations` возвращает `working_hours`, `accepting_orders`, `is_open`, `next_open_at` для вывода на странице локаций.
- В `create_order`/`mark_paid`: точка закрыта по расписанию → **HTTP 409** `LOCATION_CLOSED {next_open_at}`; на ручной паузе → **409** `LOCATION_PAUSED`.
- Итоговое условие прохождения заказа: `is_open && (remaining == null || remaining >= drinks_in_order)`.

### 5.9. Сотрудники, привязка к локации и доступ менеджера (Р13)
**Модель.** В `staff_users` добавляем `location_id` (FK locations, **nullable**). Семантика по ролям:
- `manager` — **обязан** иметь `location_id`; работает только со своей точкой;
- `super_admin` — `location_id = null` (видит все точки).

**Создание/редактирование сотрудника (супер-админ).** В `/admin/staff` форма сотрудника получает поле **«Location»** (select из активных локаций). Супер-админ прикрепляет/меняет точку менеджера. Валидация: для роли `manager` локация обязательна.

**Контроль доступа (ключевое — чтобы менеджер не трогал чужие заказы).**
- Бэкенд: во **всех** админ-эндпоинтах заказов (`GET /api/admin/orders`, `/orders/{id}`, `take`, `status`, `refund`) для роли `manager` **жёстко фильтруем/проверяем `order.location_id == staff.location_id`**. Попытка открыть/взять чужой заказ → **403 `FOREIGN_LOCATION`** (фильтрация на уровне запроса, не только UI).
- Заказ относится к точке через `order.location_id` (фиксируется при создании, Р5).
- Realtime-канал админки `ws/admin/orders` для менеджера подписывает только его `location_id`.
- Фронт `/admin/orders`: менеджер видит заголовок «Location: <name>» и список **только своей точки**, без переключателя точек; супер-админ — все точки + фильтр по локации.

**API.**
```
POST  /api/staff/managers            (… + location_id)      # super_admin
PATCH /api/staff/managers/{id}       (… + location_id)      # super_admin: сменить точку
GET   /api/admin/orders?location_id= (super_admin: фильтр; manager: игнор, всегда своя точка)
```

### 5.10. Аналитика по локациям (дашборд, Р13 + сравнение лимитов)
Супер-админ должен **сравнивать точки** и видеть, **как расходуется лимит** по каждой.
- **Дашборд `/admin` (super_admin):** разрез метрик **по локациям** — выручка, заказы, напитки; колонка/виджет **«sold today / limit / remaining / % выработки»** по каждой точке рядом для сравнения; «No limit» для точек без лимита.
- **Локация менеджера видна в аналитике:** в списках сотрудников/заказов и в разрезах указывается, **за какой точкой закреплён менеджер** (кто из персонала на какой локации).
- **Менеджер на дашборде** (если доступен) видит метрики **только своей точки**.
- **На рабочем экране заказов** (`/admin/orders`) менеджер видит мини-панель своей точки: **sold today / remaining / limit**, статус «open/closed/paused» — чтобы понимать, сколько ещё можно принять (закрывает UX-пробел).
- Данные берём из `orders.location_id` (исторично) и `location_daily_counters` (по дням). **Дашборд-эндпоинт Juicy дорабатываем:** добавляем группировку/фильтр `?location_id=` (сейчас его нет) и блок сравнения точек.

### 5.11. Управление контентом, медиа и операционные настройки в админке
> Закрывает пробелы аудита Juicy-админки: нет загрузки медиа, нет CMS контента, нет операционной паузы точки, нет экрана настроек.

**5.11.1. Контент сайта (CMS) — раздел `/admin/content` (в скоупе, не опционально).**
Управляем редактируемым контентом публичного GRABZI без релиза, через таблицу `info_blocks`:
- блоки инфо-страницы (§6): **Our Story** (текст), **Contact us** (email, телефон, **соцсети** — список ссылок), подпись/тексты раздела Location;
- произвольные текстовые блоки (например, баннер/слоган на главной).
- Поля блока: `key`, `title` (JSON i18n), `body` (richtext, JSON i18n), `sort`, `is_active`.
- API: `GET/POST/PATCH /api/admin/content`, публичный `GET /api/content?locale=`.
- **Главная (hero/секции)** в v1 — из конфигурации темы GRABZI (не редактируется из админки); если потребуется редактирование hero/баннеров — расширяем `info_blocks` блоками типа `hero` (отмечено в §12).

**5.11.2. Загрузка медиа напитков/категорий — новый аплоад (важно, в Juicy его нет).**
В Juicy медиа хранится **только как URL-строка**, файл-аплоада нет — для GRABZI добавляем:
- эндпоинт `POST /api/admin/media` (multipart) → загрузка в объектное хранилище (S3-совместимое), возвращает URL; этот URL пишется в `drinks.preview_url`/`video_url`, `drink_categories.*`, опц. `locations.image_url`.
- В редакторе напитка/категории/локации — поле загрузки файла (фото/видео-петля) вместо ручного URL (URL тоже оставляем как фолбэк).
- Видео-петли карточек GRABZI — загружаются здесь же.
> Если объектное хранилище заводить не хотим на старте — фолбэк: оставить **URL-only** (как в Juicy) и хостить медиа на CDN заказчика. Решение — §12.

**5.11.3. Операционная пауза точки (accepting_orders) — новое поле + кнопка.**
Отдельно от `is_active` (полностью скрыть точку) и от расписания часов вводим **ручную операционную паузу**:
- поле `locations.accepting_orders` (bool, default true);
- в админке (и на рабочем экране менеджера своей точки) — кнопка **«Pause orders / Resume»**;
- **итоговая доступность заказа:** `is_active && accepting_orders && schedule_open(now)`. Пауза → публично точка «Temporarily not accepting orders», заказ блокируется (тот же 409 `LOCATION_CLOSED`/отдельный `LOCATION_PAUSED`).
- Кто может: super_admin — любую точку; **менеджер — ставить паузу только своей** точке.

**5.11.4. Стоп-лист напитка по локации — осознанное исключение (Р7) + рычаги.**
Матрицу «напиток × локация» не делаем (Р7). Но фиксируем бизнес-нужду «напиток закончился»:
- **сейчас доступный рычаг:** статус напитка в каталоге (`draft|published|hidden`) — `hidden` снимает напиток с продажи **глобально** (во всех точках). Этого достаточно для общего стоп-листа.
- **по-локационный стоп-лист** (снять напиток в одной точке) — **вне базового скоупа**, помечен в §12 как расширение (потребует лёгкую таблицу `location_drink_stops`).

**5.11.5. Настройки: что в админке, что в конфиге.**
Явная граница (в Juicy экрана настроек нет — это by design):
- **В админке (данные):** локации (часы/лимит/пауза/описание/адрес/контент), сотрудники+привязка, каталог (напитки/категории/медиа), контент сайта.
- **В конфиге/env (инфраструктура):** Apple Pay вкл/выкл, OTP вкл/выкл, валюта/дефолтная TZ, ключи Stripe. Отдельный «экран настроек» в админке **не делаем** на старте (если понадобится — добавим `/admin/settings`, §12).

## 8. Данные и миграции

- **Alembic-миграции (дорабатываем схему Juicy):** `locations` (вкл. `description`, `daily_drink_limit nullable`, **`accepting_orders`**, **`image_url`**), `location_daily_counters`, `info_blocks`, `orders.location_id (nullable)`, **`staff_users.location_id (nullable, FK locations)`** (§5.9). **Новый статус заказа НЕ добавляем** — видимость после оплаты на существующем `payment_status` (§5.1). Контент-поля (`drinks.name/description`, `drink_categories.name`, `locations.name/description`, `info_blocks.*`) — JSON i18n `{en[,ar]}` (заполняем `en`; AR — §3.3).
- **Медиа-хранилище (§5.11.2):** объектное S3-совместимое хранилище для аплоада фото/видео; в БД — только URL. (Фолбэк без хранилища — URL-only, §12.)
- **Сиды GRABZI:**
  - Локации: 1–N точек с `working_hours` Пн–Сб 5:30–22:00 / Вс 10:00–18:00, `timezone=Asia/Dubai`, `daily_drink_limit` (напр. 150 — отсылка к «150 в день» из их Story; или `null` = без лимита).

  - `info_blocks`: story (из спарсенного), contacts.
- **Что в GRABZI просто не используем** (код Juicy не удаляем, на фронте GRABZI не выводим): добавки (`Addon*`, `DrinkAddon`, `OrderItemAddon`), купоны (`Coupon`), онбординг, ввод имени, корзина. Деталка напитка — опциональный модуль (Р3.2), на старте не подключаем.

## 9. Фазы и критерии приёмки (Definition of Done)

**Фаза 0 — Берём бэкенд Juicy как основу (0.5–1 д).** Поднять проект на коде Juicy (бэкенд+админка) под GRABZI; миграции-доработки и сиды GRABZI из парсера.
*DoD:* бэкенд Juicy запускается под GRABZI; миграции применяются; сиды GRABZI (локации/каталог) заливаются.

**Фаза 1 — Фронт GRABZI: UI-кит + адаптив + дешёвый i18n-задел (3–4 д).** Отдельное публичное приложение GRABZI; шрифты/токены/тема; `GrabziHeader`(desktop)/`GrabziBottomNav`(mobile); §3.2; брейкпоинты. **No-regret §3.3.A:** словарь `messages/en.json` (нет хардкода), **логические CSS везде**, контент JSON i18n. *(Полный AR §3.3.B — отдельный модуль.)*
*DoD:* главная GRABZI на desktop и mobile, подключена к бэкенду; навигация по брейкпоинту; строки через словарь; Lighthouse ≥ 90.

*DoD:* выбрал категорию → видны все её напитки со счётчиками; собрал заказ без корзины/добавок; превышение остатка → модалка и блок Pay; e2e «order page → Pay».

**Фаза 3 — Локации + оплата-then-видимость бэкенд (4–5 д).** Модели/миграции/сиды (`locations` вкл. `accepting_orders`,`location_daily_counters`); `GET /api/locations(+id)`; `schedule_open`/`is_open`(с учётом паузы)/`remaining`; **авто-логин по телефону без OTP (§4.7) + `POST /api/orders` → `payment_status="pending"` (лимит не трогает, менеджеру не виден)**; **`mark_paid` в вебхуке: списание лимита под row-lock + `payment_status="paid"`, idempotent**; авто-refund при провале; refund-декремент; уборка зависших; 409-контракты (`LOCATION_CLOSED`/`LOCATION_PAUSED`).
*DoD:* юнит-тесты остатка/часов (TZ-граница); **неоплаченный (`pending`) не виден менеджеру и не в статистике, лимит не тронут**; **тест гонки**: K параллельных оплат у точки с лимитом L не уводят `committed_drinks > L` (лишние авто-возвращаются); повторный вебхук не списывает дважды; refund возвращает лимит в тот же день; авто-логин по телефону без OTP (новый/существующий номер) корректно создаёт/находит юзера и привязывает заказ.

**Фаза 4 — Админка: локации + сотрудники + контент + медиа + аналитика (4–5 д).** В **админке Juicy** (дорабатываем): `/admin/locations` (CRUD: описание/адрес/фото/**часы по дням**/лимит-или-без/**пауза**, sold/remaining, история, `adjust-day`); **`staff_users.location_id` в `/admin/staff`**; **доступ менеджера — только заказы своей точки (фильтр + 403) + мини-панель остатка**; **дашборд: группировка `?location_id=`, сравнение лимитов, локация менеджера (§5.10)**; **`/admin/content` (CMS info_blocks §5.11.1)**; **аплоад медиа `POST /api/admin/media` (§5.11.2)**; купоны/добавки в навигации не показываем.
*DoD:* супер-админ создаёт точку (часы по дням, лимит-или-без, фото), ставит **паузу** точке, прикрепляет менеджера; **менеджер видит только свою точку (чужой заказ → 403) и остаток своей точки**; редактирует **Story/Contact/соцсети** из `/admin/content`; **загружает фото/видео напитка**; дашборд показывает sold/limit/remaining по точкам и локацию менеджера.

**Фаза 5 — Публичный UX локаций + мои заказы + инфо (3–4 д).** Выбор локации перед `/order` (остаток/часы/sold-out/closed/paused); индикатор остатка; обработка 409; WS/поллинг; **`/orders` с авто-логином по телефону (§4.4/§4.7)**; **`/profile` без купонов**; оценка без купона; `/info` (4 секции, контент из `info_blocks`).
*DoD:* нельзя заказать в закрытой/распроданной/паузнутой точке; гость по телефону подгружает историю; профиль без купонов; остаток в реальном времени; инфо-страница берёт Story/Contact из админки.

**Фаза 6 — Связка и регрессия (2–3 д).** Сквозной флоу GRABZI: локация → `/order` (гость по телефону) → оплата → `payment_status="paid"` → статус готовки → «I'm here» → оценка. Кросс-браузер desktop+mobile; регресс ролей админки.
*DoD:* зелёный e2e на desktop и mobile; регресс ролей админки; чек-лист §11 пройден.

*Ориентир: ~4–5 недель одним фронт+бэк; быстрее при параллели. Экономия заказчику — бэкенд+админка переиспользуются из Juicy, а не пишутся заново.*

## 11. Тест-чеклист (приёмка)
- [ ] Лимит: нельзя продать больше `daily_drink_limit` за день (вкл. гонку параллельных оплат).
- [ ] Часы: заказ в закрытой точке отклоняется (`LOCATION_CLOSED`), `next_open_at` корректен в TZ.
- [ ] Refund в тот же день возвращает напитки в лимит; в другой день — нет.
- [ ] Граница суток считается в TZ локации.
- [ ] **Без лимита (`null`):** заказы у такой точки не ограничиваются, но `committed_drinks` всё равно растёт для аналитики.
- [ ] **Часы по дням недели:** разное время открытия/закрытия по дням применяется; выходной день блокирует заказ; часы выводятся на публичной странице локаций.
- [ ] **Привязка сотрудника к локации:** менеджер видит/берёт заказы только своей точки; чужой заказ → 403; супер-админ видит все.
- [ ] **Аналитика:** дашборд показывает sold/limit/remaining по каждой точке и локацию менеджера; менеджер на рабочем экране видит остаток своей точки.
- [ ] **Операционная пауза точки:** «Pause orders» блокирует заказ (`LOCATION_PAUSED`) отдельно от расписания и `is_active`; менеджер ставит паузу только своей точке; «Resume» возвращает приём.
- [ ] **Контент-CMS:** Story/Contact/соцсети редактируются из `/admin/content` и сразу видны на `/info` без релиза.
- [ ] **Загрузка медиа:** фото/видео напитка/категории/локации загружаются файлом в админке (не только URL).
- [ ] **Заказ фиксирует `location_id`** (видно в истории/аналитике).
- [ ] Каталог без добавок: заказ создаётся без addon-строк; цена = base×qty; **купоны не выпускаются** (дизлайк без купона).
- [ ] **Создание заказа — одна страница:** выбор категории показывает все её напитки со счётчиками; превышение остатка → модалка + блок Pay.
- [ ] **Оплата-then-видимость:** `payment_status="pending"` скрыт от менеджера и статистики, лимит не тронут; после оплаты → `paid` + списание лимита; повторный вебхук idempotent; зависшие чистятся.
- [ ] **Авто-логин по телефону без OTP:** новый телефон → создаётся юзер + токен; существующий → находится и авторизуется; заказ привязан, `user_id` проставлен.
- [ ] **Мои заказы:** ввод телефона → авто-логин → история этого пользователя (OTP сейчас не требуется; усиление — флагом на будущее).
- [ ] **Переиспользование:** бэкенд+админка взяты из Juicy и доработаны; неиспользуемое (добавки/купоны) не удалено и не ломает работу.
- [ ] Desktop и mobile: навигация, каталог-сетки, страница заказа, выбор локации.
- [ ] EN-only публичная часть; переключатель языка скрыт.
- [ ] **AR-задел (§3.3.A):** публичные строки через словарь (нет хардкода); вёрстка на логических CSS-свойствах; контент-поля — JSON i18n. *(Полный AR/RTL — отдельный модуль, не в базовом скоупе.)*
- [ ] Остаток обновляется в реальном времени (WS) и в фолбэке (поллинг).
- [ ] Ролевой доступ админки (manager vs super_admin) сохранён.
- [ ] Сквозной флоу заказа зелёный на обеих версиях.

## 12. Открытые вопросы (с дефолтами, не блокируют)
1. `/info` vs `/about` (дефолт `/info`); контент из админки (дефолт — да, `info_blocks`).
2. Деталка напитка (Р3.2) — дефолт **выкл** (опц. модуль).
3. Перенос `is_free_treat_day` («free treat day») из GRABZI — **вне базового скоупа** (позже).
4. Количество локаций на старте и их реальные адреса/координаты/часы — уточнить у заказчика для сидов.
5. Apple Pay — включаем при готовности Stripe-аккаунта.
6. **Арабская версия (§3.3.B)** — под вопросом; в базовый скоуп не входит, отдельный модуль по подтверждению.
7. Менеджер за **несколькими** точками? Дефолт — **одна** (иначе `staff_users.location_id` → m2m `staff_locations`).
8. Где живёт код GRABZI — отдельный репозиторий (копия Juicy-бэкенда) или ветка от Juicy. Дефолт — как удобнее разработке; это проект GRABZI на базе кода Juicy (§A).
9. **OTP при входе/истории** — сейчас **выключен** (`auth_otp_enabled=false`), авто-логин по телефону без кода (§4.7). На будущее можно включить флагом (код в Juicy есть), чтобы подтверждать владение номером.
10. **`/menu` vs `/order`** — в GRABZI это **один экран** (страница создания заказа §4.2 = просмотр напитков по категориям + счётчики). Отдельного browse-only `/menu` не делаем, чтобы не дублировать; пункт «Menu» в навигации ведёт на `/order`.
11. **Где живёт фронт GRABZI** — отдельное Next.js-приложение (новая папка/проект), использует свой API-клиент к доработанному бэкенду Juicy. Juicy-фронт (`app/`) не трогаем.
12. **Медиа-хранилище (§5.11.2)** — заводим объектное хранилище (S3-совместимое) под аплоад фото/видео, **или** фолбэк URL-only (как в Juicy) с хостингом медиа на стороне заказчика. Дефолт — аплоад в хранилище (контент-менеджеру удобнее).
13. **Редактирование главной (hero/баннеры)** — в v1 hero из конфигурации темы (не из админки). Если нужно редактировать — расширяем `info_blocks` блоком `hero` (§5.11.1).
14. **По-локационный стоп-лист напитка** — вне базового скоупа (Р7). Текущий рычаг — статус напитка `hidden` (глобально). Если нужно снимать напиток в одной точке — добавить `location_drink_stops` (§5.11.4).
15. **Экран настроек `/admin/settings`** — на старте не делаем; инфраструктурные параметры в env (§5.11.5). Добавим, если заказчику нужно менять их без разработчика.
16. **TRN/налоговые реквизиты точки** (для чеков) — в Juicy-моке у аутлета был TRN; если нужны в чеках GRABZI, добавить `locations.trn` (сейчас не закладываем).
