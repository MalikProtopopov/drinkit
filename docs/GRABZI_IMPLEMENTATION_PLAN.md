# GRABZI — план реализации фронтенда на базе функционала Juicy

> Цель: взять **структуру и бренд GRABZI**, надеть на неё **функционал, отрисованный в прототипах Juicy** (наш основной проект), и добавить **новый функционал — мультилокационные дневные лимиты напитков**, которого нет ни в Juicy, ни в текущем GRABZI.
>
> Документ собран по результатам парсинга `grabzi.ae` (Shopify) и `order.grabzi.ae` (React/Supabase SPA) и аудита проекта Juicy (`app/` — Next.js, `backend/` — FastAPI).

---

## 0. TL;DR — что делаем

1. **Публичный сайт** остаётся в структуре GRABZI (главная + инфо + заказ), но переезжает на наш стек и получает функционал Juicy (каталог, корзина, кастомизация, статусы заказа, «я на месте», профиль, купоны).
2. **Дизайн публичной части** — бренд GRABZI (шрифт Grenadine MVB, цвет `#c44429`, эмодзи-нейминг напитков), а не дизайн Juicy. Админка остаётся на дизайне Juicy.
3. **Новый сквозной функционал — Локации с дневным лимитом:**
   - в админке новый раздел **«Локации»**: создать локацию, задать дневной лимит напитков;
   - на публичном сайте: выбор локации, отображение «осталось N из M напитков сегодня»;
   - при создании заказа — **атомарная проверка**: количество напитков в заказе не превышает остаток лимита локации, иначе ошибка.
4. **Инфо-страницы объединяются в одну** (`Location + Shop Story + Working hours + Contact us` → один экран с якорными секциями).

**Зафиксированные решения (§7):** лимит списывается при **оплате**, считается в **напитках**; каталог **одинаков для всех локаций**; публичный UI — **только английский**; **вне рабочих часов локации заказ для неё недоступен**.

## 4. НОВЫЙ ФУНКЦИОНАЛ: Локации с дневным лимитом напитков

Самая важная часть. Ниже — модель, API, UI и **корректная обработка гонки** при проверке лимита.

### 4.1. Модель данных (backend, новые таблицы)
**`locations`**
| Поле | Тип | Назначение |
|---|---|---|
| `id` | PK | |
| `name` | JSON i18n | название точки |
| `address` | str | адрес |
| `coordinates` | str/json | гео (для сортировки по расстоянию) |
| `working_hours` | json | часы работы |
| `timezone` | str | TZ для границы «дня» (по умолчанию `Asia/Dubai`) |
| `daily_drink_limit` | int | **сколько напитков можно заказать за день** |
| `color` | str | фон карточки |
| `is_active` | bool | вкл/выкл точку |
| `sort` | int | порядок |

**`location_daily_counters`** (счётчик-на-день, для атомарной проверки)
| Поле | Тип | Назначение |
|---|---|---|
| `id` | PK | |
| `location_id` | FK locations | |
| `date` | date | день (в TZ локации) |
| `committed_drinks` | int | сколько напитков уже «занято» оплаченными/принятыми заказами |
| уникальный индекс | (`location_id`,`date`) | одна строка на точку-день |

**`orders`** — добавить `location_id` (FK locations, NOT NULL для новых заказов).

> Почему отдельный счётчик, а не `COUNT(*)` по заказам: атомарность и отсутствие гонок. Считать на лету тоже можно, но тогда блокировку всё равно надо вешать — счётчик делает её явной и дешёвой.

### 4.2. Логика остатка
```
sold_today(location)     = location_daily_counters[location, today].committed_drinks
remaining(location)      = max(0, location.daily_drink_limit - sold_today(location))
is_sold_out(location)    = remaining(location) == 0
```
`today` берётся в `location.timezone` (граница суток локальная, не UTC).

### 4.3. Проверка при создании заказа (анти-oversell)
В транзакции `create_order`:
1. `SELECT ... FOR UPDATE` строки `location_daily_counters[location_id, today]` (создать с 0, если нет) — **блокируем строку**.
2. `drinks_in_order = sum(item.quantity)`.
3. Если `committed_drinks + drinks_in_order > daily_drink_limit` → откат, **HTTP 409** с телом:
   ```json
   { "code": "LOCATION_LIMIT_EXCEEDED", "remaining": <N>, "requested": <K> }
   ```
4. Иначе `committed_drinks += drinks_in_order`, коммит вместе с заказом.

**Когда «занимать» лимит** — выбрать политику (см. §7 «Решения»):
- **A) при создании заказа** (резерв) — проще, но неоплаченные заказы «съедают» лимит → нужен таймаут-возврат резерва;
- **B) при оплате** (`payment_status=paid`) — лимит занимают только реальные продажи; до оплаты показываем остаток без резерва (риск редкого овэрселла на параллельных оплатах, но проверка в момент `mark_paid` его закрывает).
> Рекомендация: **B (списываем при оплате)** + повторная проверка лимита в `mark_paid` под тем же row-lock; на этапе чекаута показываем «осталось N», при оплате — финальная валидация. Это совпадает с тем, как у GRABZI считается `cups_sold` по факту.

### 4.4. API
**Публичное**
- `GET /api/locations` → список активных локаций с `remaining_today`, `is_sold_out`, адресом, часами, координатами.
- `GET /api/locations/{id}` → детально + остаток.
- `POST /api/orders` → теперь требует `location_id`; валидация лимита (§4.3); при превышении — 409 `LOCATION_LIMIT_EXCEEDED`.

**Админка (super_admin)**
- `GET /api/admin/locations` — список с лимитом, продано/остаток за сегодня.
- `POST /api/admin/locations` — создать (имя, адрес, часы, TZ, `daily_drink_limit`).
- `PATCH /api/admin/locations/{id}` — изменить (в т.ч. лимит, активность).
- `GET /api/admin/locations/{id}/daily?date=` — продано/остаток за день (история).
- `POST /api/admin/locations/{id}/reset-day` *(опц.)* — ручной сброс/корректировка счётчика дня.

### 4.7. Рабочие часы локации блокируют заказ (новое требование)
Каждая локация имеет `working_hours` (+ `timezone`). Вне окна работы заказ для этой точки **недоступен** — аналог `is_shop_open` у GRABZI, но на уровне локации.

- **Бэкенд:** функция `is_location_open(location, now)` по `working_hours`/`timezone`. В `GET /api/locations` отдаём `is_open: bool` и `next_open_at`. В `create_order`/`mark_paid` — если точка закрыта, **HTTP 409** `LOCATION_CLOSED` `{ next_open_at }`.
- **Формат `working_hours`:** по дням недели интервалы (напр. `{"mon-sat": "05:30-22:00", "sun": "10:00-18:00"}` — берём из спарсенных часов GRABZI), редактируется в админке.
- **UI:** закрытая локация в выборе помечается «Closed · opens at HH:MM», кнопка заказа недоступна; если точка закрылась пока пользователь в корзине — блок оплаты + подсказка.
- **Связка с лимитом:** заказ проходит только если `is_open && remaining >= drinks_in_order`.

### 4.5. UI — публичная часть
- **Экран выбора локации** (`/outlets` → реальный): карточки точек, на каждой бейдж **«осталось N из M сегодня»**; `is_sold_out` → карточка «Sold out», заказ недоступен.
- **Кнопка «Создать заказ»** на главной → если локация не выбрана, ведёт на выбор локации, затем в каталог.
- **Шапка каталога/корзины:** постоянный индикатор остатка выбранной локации.
- **Корзина/чекаут:** если `кол-во напитков в корзине > remaining` — блок кнопки оплаты + подсказка «В этой локации осталось N напитков».
- **Ошибка 409 при оплате:** тост/экран «Лимит локации исчерпан, осталось N» + предложение сменить локацию/уменьшить заказ.
- Реалтайм-обновление остатка (WebSocket, как у заказов) — опционально, иначе поллинг.

### 4.6. UI — админка (новый раздел «Локации»)
- `/admin/locations` — список: название, адрес, лимит/день, **продано сегодня / остаток**, статус.
- Модалка «Новая локация»: имя (RU/AR), адрес, часы, TZ, дневной лимит.
- `/admin/locations/[id]` — карточка: редактирование, график продано/остаток по дням, кнопка корректировки дня.
- Привязка: какие напитки доступны в локации (матрица «напиток × локация» — переиспользуем мок-идею аутлетов Juicy, но на реальном бэке) — *опционально, фаза 2*.

## 7. Принятые решения (зафиксировано)

1. **Момент списания лимита — при ОПЛАТЕ.** Лимит занимают только оплаченные заказы (как `cups_sold` у GRABZI). До оплаты остаток показывается без резерва; финальная проверка под row-lock в `mark_paid`.
2. **Лимит считается в НАПИТКАХ** (`sum(item.quantity)`).
3. **Каталог одинаков для всех локаций** — все напитки везде. Матрица «напиток × локация» **не делаем** (различается только остаток лимита).
4. **Язык публичной части — ТОЛЬКО английский.** Переключатель RU/AR на публичной части убираем (i18n-инфраструктура Juicy остаётся в БД, но публичный UI — EN). Админка — без изменений.
