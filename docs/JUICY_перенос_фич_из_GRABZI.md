# Перенос фич из GRABZI в Juicy — что добавить и куда интегрировать

> **Контекст.** GRABZI и Juicy используют **общий backend** (одна кодовая база; ветка `grabzi` = `main`/Juicy
> + 20 коммитов). В рамках GRABZI реализованы несколько улучшений, которые **уже заложены в спеке Juicy**
> (`docs/BACKEND.md`), но в прототипе Juicy (`app/`) не собраны. Перенос = «включить» их в Juicy, переиспользовав
> готовый backend-код. **Локации/мультиточечность и дневной лимит — НЕ переносим** (у Juicy уже есть свои
> «точки/outlets»; берём только удобство, а не концепцию сети). Дата: 2026-06-11.

## 1. Список переносимых обновлений (для заказчика — коротко)

| # | Обновление | Что даёт бизнесу | Приоритет |
|---|---|---|---|
| F1 | **Время работы для приёма заказов** (расписание + пауза) | заказы принимаются только в рабочие часы точки; менеджер может вручную поставить приём на паузу (наплыв/перерыв) | **Must** |
| F2 | **Drag-and-drop загрузка медиа в админке** (S3/MinIO) | фото/видео/постеры грузятся перетаскиванием прямо в карточку, без ручной вставки URL | **Must** |
| F3 | **Стоп-лист наличия по точке** (быстрый «нет в наличии») | кончился ингредиент — товар одним кликом скрывается на точке, клиент не закажет недоступное | **Should** |
| F4 | **Глобальные настройки сети** (без программиста) | валюта/налог/часовой пояс/контакты поддержки/флаги модулей правятся в админке | **Should** |
| F5 | **Контент-страницы (CMS)** — «О нас»/«Контакты»/«Часы»/FAQ | владелец сам редактирует тексты публичных страниц | **Could** |
| F6 | **Оффлайн-баннер + устойчивость к сети** | при потере связи клиент видит понятный баннер, а не «белый экран» | **Could** |
| F7 | **Realtime-обновления заказов** (WebSocket + polling-fallback) | статусы заказов и очередь обновляются мгновенно, без перезагрузки | **Could** |

> F1 и F2 — то, что вы назвали в первую очередь (рабочие часы приёма заказов и drag-drop медиа).

## 2. Карта интеграции — куда и как (для разработчика)

**Канон-источник (ветка `grabzi`):**
- расписание/приём заказов — `backend/app/services/location_service.py` (`is_open` / `business_date` / расписание), `backend/app/routers/admin_locations.py` (pause / stop-list);
- медиа — `backend/app/services/storage.py` + `backend/app/routers/admin_media.py` (`POST /api/admin/media` → `{key, url}`, S3/MinIO);
- настройки/контент — `backend/app/models/locations.py` (`AppSetting`, `InfoBlock`), `backend/app/routers/admin_settings.py`, `routers/content.py`;
- realtime — `backend/app/routers/ws.py` + `backend/app/core/pubsub.py`;
- фронт-референс — `grabzi-web/` (оффлайн-баннер `src/components/OfflineBanner.tsx`).

### F1. Время работы для приёма заказов (расписание + пауза)
- **Куда в Juicy:** карточка точки `app/app/admin/outlets/[id]/page.tsx` — вкладка **«Расписание»** (сейчас
  это placeholder, см. `t === "hours"`). Спека уже предусматривает (`docs/BACKEND.md §2.1 Outlet` —
  рабочие часы). Гейт приёма заказов — на оформлении (`app/app/checkout` / `app/lib/api.ts`).
- **Как:** перенести из grabzi `working_hours` (JSON по дням недели + `timezone`) и логику
  `location_service.is_open()` (учёт перехода через полночь). Добавить поле `accepting_orders` (ручная
  пауза) и эндпоинт `pause/resume`. На клиенте — бейдж «Открыто/Закрыто/Пауза» и блокировка кнопки заказа
  с понятным сообщением (`LOCATION_CLOSED` / `LOCATION_PAUSED` — коды уже в `core/errors.py`).
- **Объём:** Бэк (модель часов + is_open + pause) — реюз grabzi; фронт — вкладка «Расписание» + гейт.

### F2. Drag-and-drop загрузка медиа (S3/MinIO)
- **Куда в Juicy:** карточки каталога, где сейчас **вставка URL вручную**:
  `app/app/admin/catalog/products/[slug]/page.tsx` (`previewUrl` — text input),
  `app/app/admin/catalog/categories/page.tsx` (`photoUrl` — «фото URL»), addons. Общий UI —
  `app/components/admin/AdminUI.tsx`.
- **Как:** подключить backend-эндпоинт `POST /api/admin/media` (есть в grabzi, S3/MinIO; спека Juicy
  `BACKEND.md §29` это и требует). Добавить компонент **`MediaDropzone`** (drag-drop зона + клик-выбор +
  превью + прогресс) в `app/components/admin/`; в карточках заменить `<input value=URL>` на дропзону,
  которая загружает файл и подставляет вернувшийся `url`. Оставить ввод URL как фолбэк.
- **Объём:** Бэк — реюз (`admin_media.py` + `storage.py`); фронт — новый компонент дропзоны + замена в 3 карточках.

### F3. Стоп-лист наличия по точке
- **Куда в Juicy:** вкладка «Меню точки» карточки точки (`outlets/[id]`, `t === "menu"`) и/или карточка
  товара; спека — `BACKEND.md §2.7 OutletInventory`, поле `inStock` per-outlet.
- **Как:** перенести grabzi `LocationDrinkStop` (наличие = отсутствие записи стопа) + проверку при создании
  заказа (`DRINK_UNAVAILABLE_AT_LOCATION`). На фронте — переключатель «В наличии / Стоп» у позиции, у клиента
  бейдж «Sold out» и блок заказа.
- **Объём:** Бэк — реюз; фронт — тумблер в меню точки + бейдж у клиента.

### F4. Глобальные настройки сети (AppSetting)
- **Куда в Juicy:** новый раздел админки `app/app/admin/settings/page.tsx` (в `app/components/admin/AdminShell.tsx`
  — пункт меню).
- **Как:** перенести `AppSetting` (модель + `settings_registry`) и `admin_settings.py` (GET/PATCH с
  валидацией по реестру). UI — форма по реестру (валюта/TZ/локали/контакты/флаги опц. модулей).
- **Объём:** Бэк — реюз; фронт — экран настроек.

### F5. Контент-страницы (CMS / InfoBlock)
- **Куда в Juicy:** новый раздел админки `app/app/admin/content/` + публичная страница (например `app/app/info`).
- **Как:** перенести `InfoBlock` + `routers/content.py` (публичный `GET /api/content`) и admin-CRUD. UI —
  список блоков (заголовок + richtext, i18n, сортировка, активность) + рендер на публичной странице с фолбэком языка.
- **Объём:** Бэк — реюз; фронт — экран контента + публичная страница.

### F6. Оффлайн-баннер + устойчивость к сети
- **Куда в Juicy:** глобальный layout `app/app/layout.tsx` (как в grabzi).
- **Как:** перенести компонент `OfflineBanner` (слушает `navigator.onLine`), показывать баннер на всех
  страницах; добавить дружелюбные состояния ошибок загрузки с «Повторить».
- **Объём:** фронт — 1 компонент + подключение в layout.

### F7. Realtime-обновления (WebSocket + polling-fallback)
- **Куда в Juicy:** экран статуса заказа клиента (`app/app/orders/[id]`) и админ-заказы/KDS.
- **Как:** перенести `routers/ws.py` (каналы `order:{id}`, `admin:orders`) + `core/pubsub.py`; на фронте —
  WS-подписка с fallback на polling (как в grabzi-web `orders/[id]` и `admin/kitchen`).
- **Объём:** Бэк — реюз; фронт — WS-клиент + fallback.

## 3. Что НЕ переносим (сознательно)
- **Концепцию мультиточечной сети «локаций»** как таковую — у Juicy уже есть свои `outlets`; берём только
  удобство (часы/пауза/наличие/медиа), а не новую модель сети.
- **Дневной лимит порций «come early — we don't make more»** — это часть бизнес-модели GRABZI (дефицит),
  Juicy её не требует.
- **Curbside-выдачу по номеру машины** — специфична для drive-through GRABZI.
- **Редизайн под grabzi.ae** — у Juicy свой бренд.

## 4. Итог (для заказчика)
Все переносимые фичи (F1–F7) уже **есть на стороне сервера** (общий backend), а в Juicy-спеке (`BACKEND.md`)
они и были запланированы — нужно лишь **собрать админ-UI и подключить**. Поэтому в плане запуска (см.
`JUICY_план_запуска_8_дней.md`) они встроены в общий поток как обычные этапы, без отдельного «портирования».
