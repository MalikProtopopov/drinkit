## Продукт и бренд
- **GRABZI** — mobile-web сервис предзаказа спешелти-кофе по модели drive-through (curbside pickup,
  выдача по номеру машины). Слоган-концепция: «Come early — we don't make more» (дневной лимит).
- Рынок ОАЭ (Dubai), бренд grabzi.ae, англоязычный UI (EN-first), инфраструктура AR/RTL заложена (V1.1).
- Модель выручки: прямые продажи кофе через mobile-web, оплата картой (Stripe), без агрегаторов доставки.

## Персонал (сид) и роли

## Статусы и деньги
- `Order.status`: `new → in_progress → ready → completed` (+ `refund`).
- `payment_status`: `pending → paid` (`failed` / `refunded`). `Payment.status`: `pending/succeeded/failed/refunded`.
- `Coupon.status`: `active / used / void`. `Drink.status`: `draft / published / hidden`.
- `arrived_at` — **независимый флаг** «клиент на месте», НЕ часть цепочки статусов.
- Деньги: **AED**. Расчёт итога: `order.total = subtotal − coupon_discount` (см. `order_flow.py`).
  ⚠️ **VAT 5 % НЕ реализован** в коде (нет налоговой строки; цена = финальная). В ТЗ описать VAT 5 % как
  **требование/решение** (модель «включён в цену» по умолчанию) и пометить статус ⚠️ «не реализовано / к
  настройке». Пример чека приводить с явной пометкой допущения о VAT.

## Дневной лимит (ядро бизнес-модели)
- `LocationDailyCounter(location_id, business_date, committed_drinks)` — считает **напитки** (сумму
  количеств), не заказы. Бизнес-день — в TZ точки; сбрасывается в полночь по времени точки.
- Проверка: мягкая при наборе заказа; **жёсткая при оплате под `FOR UPDATE`** (`order_flow.py`) —
  гарантирует продажу ровно последней порции одному покупателю (защита от гонки).
- Возврат (`refund`) в тот же бизнес-день декрементирует счётчик (не ниже 0).
- Клиент видит «TODAY'S LIMIT — осталось N» / «Sold out for today»; кухня — прогресс «продано X / лимит Y».

## Купоны / рейтинг
- После заказа клиент ставит 👍 или 👎. **👎 (dislike) выдаёт купон** (компенсация, скидка на следующий
  заказ); 👍 не выдаёт. Один рейтинг на заказ. Купон применяется к **одной** позиции следующего заказа.
- Номинал/срок купона в коде не зафиксированы жёстко → вынести как ❓ открытый вопрос (это расход бизнеса).
- Супер-админ может аннулировать активный купон (`void`).

## Стоп-лист и статус точки (операционные рычаги)
- `LocationDrinkStop` (existence-based): напиток помечен недоступным на ТОЧКЕ → клиент видит «Sold out»,
  заказать нельзя; снять — обратным действием. Проверяется при создании заказа.
- Пауза точки (`accepting_orders=false`) — менеджер/супер-админ временно останавливает приём заказов.
- Все операции пишут аудит: `LocationStatusEvent`, `OrderEvent`.

## Интеграции (статус)
- **Stripe** — реальный; без `STRIPE_SECRET_KEY` → **mock**: оплата сразу `paid`, редирект `/orders/{id}?paid=1`.
  ⚠️ В mock реальные деньги НЕ списываются — для прода нужен Stripe merchant + TRN.
- **SMS-OTP** — **заглушка**: провайдер не выбран; вход по телефону с пустым кодом (`code:""`), dev-режим
  вернул бы код. ⚠️ Сейчас вход без проверки кода — для прода подключить SMS-провайдера.
- **Redis** — опционально (in-memory fallback). **WebSocket** — реальный (asyncio). **S3/MinIO** — реальный.
- **JWT** — HS256, TTL 30 дней; `core/security.py`. Пароли персонала — pbkdf2_sha256.

## Авторизация / токены / стор (фронт)
- Клиент: телефон + OTP (сейчас вход без кода). Токен в `localStorage['grabzi_token']`.
- Персонал: email + пароль. Токен в `localStorage['grabzi_staff_token']`.
- Черновик заказа (Zustand persist): `localStorage['grabzi-draft']` = `{state:{locationId, items}, version:1}`.
- Телефон — канон-валидация бэка **`^\+\d{9,15}$`** (фронт мягче `^\+?\d{7,15}$` ⚠️ расхождение).
- API base: `http://localhost:8000`. Заголовок защищённых вызовов: `Authorization: Bearer <token>`.

## ⚠️ Известные дефекты/расхождения прототипа (внести в реестр открытых вопросов / противоречий)
1. **Kitchen ↔ location-status контракт:** фронт `LocStatusSchema` ждёт `name: string`, бэкенд отдаёт
   `name: {"en": …}` (i18n-объект) → zod-парс падает → ложное «Session expired». Для скриншота имя
   нормализовано на лету (см. примечание к ADM-M). Реальный дефект — описать в Часть IX.
2. **Super-admin на экране кухни:** `/api/admin/location-status` для super_admin (без точки) отдаёт
   `{scope:"all", locations:[]}`, которого схема кухни не ожидает → кухня рассчитана на **менеджера**.
3. **VAT 5 %** не реализован (см. выше).
4. **Телефон-regex** фронт/бэк расходятся (см. выше).

**Для заказчика** (Проблема / Выгода / Что даёт бизнесу) → **Приоритет (MoSCoW)** → **Как <роль>…** →
**Как это работает (простыми словами)** (без кода) → **AC** (нумерованные, с визуальным+a11y+ролевым) →
**Экран и стиль** (код, маршрут, 📷 скрин, токены/компоненты) → *— Технические детали —* **Бизнес-логика
(под капотом)** → **API** → **Что видит другая сторона** → **Трассировка** → **Метки** (✅/🔧/🚧/🧩/❓/⚠️).
