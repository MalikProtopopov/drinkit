## I.1 Технологический стек

> **Для заказчика.** Продукт состоит из двух частей: «витрина», которую видит покупатель в браузере
> телефона, и «сервер» — мозг системы, где хранятся заказы, цены, точки и считается дневной лимит.
> Витрина быстрая и не требует установки приложения (это веб-страница). Сервер устроен так, чтобы
> запускаться и на простом «ноутбучном» хранилище данных для демонстрации, и на «промышленной» базе
> для реальной работы — без переписывания.

### Как это работает (простыми словами)

Покупатель открывает сайт в браузере телефона — это и есть всё «приложение», ничего скачивать не
нужно. Когда он собирает заказ или платит, браузер обращается к серверу, а сервер хранит данные,
проверяет правила (открыта ли точка, есть ли свободные порции на сегодня) и мгновенно сообщает кухне
о новом заказе. Кухня видит очередь на планшете и обновляется в реальном времени — без нажатия
«обновить».

### Технические детали (для разработки)

| Слой | Технология | Назначение | Канон-источник |
|---|---|---|---|
| Фронтенд | **Next.js 16 / React 19** (App Router, TypeScript) | клиентское mobile-web + экран кухни (KDS) | `grabzi-web/` |
| Состояние фронта | **Zustand** (persist в `localStorage`) | черновик заказа `grabzi-draft` | `grabzi-web/src/app/order/` |
| Бэкенд | **FastAPI** (Python), **Pydantic v2** (`pydantic-settings`) | REST API + WebSocket + вебхук Stripe | `backend/app/main.py`, `backend/app/core/config.py` |
| ORM | **SQLAlchemy** | доменная модель, транзакции, `FOR UPDATE` | `backend/app/models/**` |
| База данных | **SQLite** (dev, файл) → **PostgreSQL 16** (прод) | хранение данных | `config.py` (`database_url`), `docker-compose.yml` (`postgresql+psycopg`) |
| Кэш / шина | **Redis 7** (опционально) | pub/sub для realtime, хранение OTP | `config.py` (`redis_url`), `core/pubsub.py` |
| Realtime | **WebSocket** (asyncio, in-process) | пуш статусов заказа клиенту и кухне | `routers/ws.py`, `core/pubsub.py` |
| Хранилище медиа | **S3 / MinIO** | картинки каталога/CMS | `services/storage.py`, `routers/admin_media.py` |
| Миграции (прод) | **Alembic** | `alembic upgrade head` при старте контейнера | `docker-compose.yml` |
| Платежи | **Stripe Checkout** (hosted) + mock-режим | приём оплаты | `routers/payments.py` |

**Принцип «запускается без инфраструктуры».** По умолчанию (`config.py`) база — SQLite-файл
(`database_url = "sqlite:///./juicy.db"`; ⚠️ имя файла — наследие прежнего прототипа Juicy, в проде
переопределяется через `DATABASE_URL`), Redis отсутствует (`redis_url = None` → in-memory fallback
для pub/sub и OTP), Stripe и S3 не настроены (mock / URL-фолбэк). В проде всё переключается через
переменные окружения **без правок кода** (`docker-compose.yml`).

**Две формы запуска (канон — `docker-compose.yml`):**
- **dev без Docker** — SQLite + uvicorn + in-memory pubsub; быстрый старт, схема через `create_all`;
- **прод/полный стек (Docker)** — PostgreSQL + Redis + MinIO; схема через Alembic; команда контейнера
  `alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port 8000`.

📝 *Версии Next.js 16 / React 19 — по описанию проекта; точные patch-версии сверять с
`grabzi-web/package.json` при сборке релиза.*

## I.2 Авторизация и токены

> **Для заказчика.** У продукта два разных входа. **Покупатель** не заводит пароль — он входит по
> номеру телефона (в будущем — с SMS-кодом), и все его заказы привязываются к этому номеру.
> **Сотрудник** (бариста-менеджер или администратор сети) входит по рабочей почте и паролю. Это
> разделение защищает данные сети: у сотрудника точки нет доступа к деньгам и каталогу всей сети.

### Как это работает (простыми словами)

Покупатель указывает телефон — система выдаёт ему «пропуск» (он хранится в браузере), по которому его
узнают при следующем заходе и показывают его заказы. Сейчас вход происходит без SMS-кода (см. статус
интеграций, §I.3) — это временная настройка, не дефект. Сотрудник входит по почте и паролю и получает
свой «пропуск» с указанием роли: бариста видит только свою точку, администратор — всю сеть. Пропуск
действует 30 дней; по истечении нужно войти заново.

### Технические детали (для разработки)

| Аспект | Клиент (PUB-G/PUB-A) | Персонал (ADM-M/ADM-S) | Канон |
|---|---|---|---|
| Метод входа | телефон + OTP (`/api/auth/request-code` → `/api/auth/verify`) | email + пароль (`/api/staff/login`) | `routers/auth.py`, `routers/staff.py` |
| Хранение токена (фронт) | `localStorage['grabzi_token']` | `localStorage['grabzi_staff_token']` | спина §«Авторизация/токены» |
| Тип токена | **JWT**, `kind="customer"` | **JWT**, `kind="staff"`, `role` в payload | `core/security.py` |
| Заголовок защищённых вызовов | `Authorization: Bearer <token>` | `Authorization: Bearer <token>` | `core/security.py` (`HTTPBearer`) |

**JWT (канон — `core/security.py` + `config.py`):**
- алгоритм **HS256** (`jwt_alg = "HS256"`); секрет `jwt_secret` (dev-дефолт `"dev-secret-change-me"` —
  ⚠️ обязательно переопределить в проде через `JWT_SECRET`);
- срок жизни **30 дней** (`jwt_ttl_hours = 24 * 30 = 720` ч);
- payload: `{sub, kind, role, exp}`; невалидный/просроченный токен → `401 AUTH_REQUIRED`.

**Хэш паролей персонала:** **pbkdf2_sha256** (`passlib`, `core/security.py`) — pure-python, без
нативных зависимостей; bcrypt сознательно не используется.

**Ролевые зависимости (FastAPI Depends, `core/security.py`):**
- `get_current_user` — клиент (`kind="customer"`), иначе `403 FORBIDDEN`;
- `get_current_staff` — персонал (`kind="staff"`); отключённый (`disabled=true`) → `401`;
- `require_super_admin` — только `role="super_admin"`;
- `require_manager_or_super` — `manager` или `super_admin`;
- `manager_scope_location(staff)` — менеджер с `location_id` ограничен своей точкой; super-admin или
  менеджер без точки → без ограничения (см. ролевой скоуп в Части V/VI).

**Сиды учёток (см. спину):** `admin@grabzi.ae / grabzi-admin` (super_admin, `location_id=null`);
`barista@grabzi.ae / grabzi-barista` (manager, `location_id=1`).

⚠️ **Расхождение фронт/бэк по валидации телефона** (канон — бэк): бэкенд требует `^\+\d{9,15}$`
(`routers/auth.py`, `PhoneIn.phone`, ведущий `+` обязателен, 9–15 цифр); фронт использует более мягкое
`^\+?\d{7,15}$` (`grabzi-web/src/app/order/page.tsx:68`, `orders/page.tsx:32`). Привести к единому
правилу `^\+\d{9,15}$` — открытый вопрос (см. Часть IX).

## I.3 Статус интеграций / mock-режим

> **Для заказчика.** Некоторые внешние сервисы (приём денег, отправка SMS) в собранном прототипе
> работают в «режиме заглушки» — это **настройка, а не поломка**. Продукт целиком готов их
> использовать; чтобы включить их «по-настоящему», нужно подключить аккаунты провайдеров. Ниже
> честно показано, что работает реально, а что ждёт подключения — со «светофором».

### I.3.1 Светофор интеграций

| Интеграция | Статус | Что это значит | Канон |
|---|---|---|---|
| **WebSocket (realtime)** | 🟢 реальный | статусы заказа летят клиенту и кухне мгновенно (asyncio, in-process) | `routers/ws.py`, `core/pubsub.py` |
| **S3 / MinIO (медиа)** | 🟢 реальный | загрузка картинок работает; без `s3_bucket` — фолбэк на URL-строки | `services/storage.py`, `config.py` |
| **JWT-авторизация** | 🟢 реальная | пропуски подписываются и проверяются (HS256) | `core/security.py` |
| **Redis** | 🟡 опционально | при `redis_url=None` — in-memory fallback (pub/sub, OTP); для нескольких процессов нужен Redis | `config.py`, `core/pubsub.py` |
| **Stripe (оплата)** | 🔧 mock без ключа | без `STRIPE_SECRET_KEY` оплата подтверждается сразу, **деньги не списываются** | `routers/payments.py` |
| **SMS-OTP (вход)** | 🔧 заглушка | `auth_otp_enabled=false` → вход по телефону без кода; провайдер не выбран | `config.py`, `routers/auth.py` |

### I.3.2 Что значит режим заглушки для запуска (бизнес-перевод)

> Подробный сквозной разбор оплаты — см. Часть VII (процесс «Оплата»); входа — Часть VII (процесс
> «OTP»). Здесь — деловой перевод «что это значит для запуска».

**Stripe-mock (🔧)**

| Колонка | Содержание |
|---|---|
| **Что сейчас** | Оплата **имитируется**: заказ моментально помечается оплаченным, покупателя возвращает на страницу заказа (`/orders/{id}?paid=1`), но реальные деньги с карты **не списываются**. |
| **Что это значит для запуска** | Принимать **реальные деньги нельзя**. Для демонстраций, тестов и приёмки — пригодно: весь поток заказа и кухни работает «как настоящий». |
| **Что нужно от заказчика** | Аккаунт **Stripe (merchant)** + налоговый номер **TRN** (ОАЭ), ключи `STRIPE_SECRET_KEY` и `STRIPE_WEBHOOK_SECRET`. |
| **Усилия на подключение** | 📝 ориентир: настройка ключей и проверка вебхука — порядка 0.5–1 дня разработки + время на открытие/верификацию аккаунта Stripe на стороне заказчика (внешний срок). Код за флагом уже есть (`routers/payments.py`), переписывать не нужно. |

**OTP-заглушка (🔧)**

| Колонка | Содержание |
|---|---|
| **Что сейчас** | Вход **без SMS-кода**: `auth_otp_enabled=false`, фронт сразу вызывает подтверждение с пустым кодом и пускает по номеру телефона. SMS-провайдер не выбран. В dev-режиме «код» фиксированный (`otp_dev_code="1836"`) и вернулся бы в ответе API. |
| **Что это значит для запуска** | Принимать заказы можно (телефон = контакт для выдачи; покупатель мотивирован указать верный — заказ уже оплачен). Но **подлинность номера не проверяется** — для публичного запуска это риск (чужой номер, спам). |
| **Что нужно от заказчика** | Выбранный **SMS-провайдер** (например, региональный для ОАЭ) + учётные данные. Включение — флагом `AUTH_OTP_ENABLED=true` **без правок кода** (адаптер подключается в `services/sms.py`). |
| **Усилия на подключение** | 📝 ориентир: адаптер провайдера + включение проверки кода в `/verify` — порядка 1–2 дней разработки (логика верификации в `routers/auth.py` уже заложена за флагом). |

> **Готово к публичному приёму реальных денег, когда подключены:** Stripe merchant + TRN
> (`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`) **и** выбранный SMS-провайдер (`AUTH_OTP_ENABLED=true`).
> Тон: это **конфигурация, а не доработка продукта**. Связь со Scope — см. Часть 0 (§0.2) и допущения
> (§I.4); бизнес-правила — см. Часть IX (каталог BR).

## I.5 Нефункциональные требования (NFR)

> **Для заказчика.** Это требования «как хорошо», а не «что делает» система: насколько быстро,
> насколько надёжно, на каких устройствах. Каждое сформулировано как **проверяемый порог** — его
> можно подтвердить замером, а не на словах.

| ID | Категория | Измеримый порог | Способ проверки |
|---|---|---|---|
| NFR-01 | Производительность (API) | p95 времени ответа ключевых GET-эндпоинтов (`/api/locations`, `/api/drinks`, `/api/orders/{id}`) ≤ **300 мс** при типичной нагрузке точки | нагрузочный замер (k6/Locust) на прод-подобной БД |
| NFR-02 | Производительность (оплата) | p95 `POST /api/payments/checkout-session` (включая `FOR UPDATE` на счётчике) ≤ **800 мс** | нагрузочный замер с конкурентными заказами |
| NFR-03 | Realtime | задержка WS-пуша смены статуса (кухня → клиент) ≤ **1 с** в 95 % случаев; fallback polling клиента **20 с**, кухни **10 с** | замер end-to-end (см. Часть VII) |
| NFR-04 | Производительность (фронт) | TTI экрана `/menu` на эмуляции **3G** ≤ **3 с** | Lighthouse mobile (throttling Fast 3G) |
| NFR-05 | Ёмкость | устойчивая обработка ≥ **20** одновременных оформлений заказа на точку без потери корректности лимита | стресс-тест гонки за последнюю порцию (есть тест в S9) |
| NFR-06 | Ёмкость | дневной лимит точки конфигурируем (сид `daily_drink_limit=150`); система корректна при любом значении и при `null` («без лимита») | модульные тесты лимита |
| NFR-07 | Доступность (uptime) | целевой uptime прод-сервиса ≥ **99.5 %**/мес; окно планового обслуживания — вне утреннего пика точки | мониторинг (uptime-чек) |
| NFR-08 | Надёжность (оплата) | вебхук оплаты **идемпотентен**: повторная доставка `checkout.session.completed` не двойно списывает лимит (`payment_status != "paid"` гард) | тест повторной доставки события |
| NFR-09 | Надёжность (RTO/RPO) | RPO ≤ **5 мин** (бэкап БД), RTO ≤ **1 ч** на восстановление прод-инстанса | регламент бэкапов + учебное восстановление |
| NFR-10 | Целостность лимита | продажа последней порции — ровно одному покупателю (пессимистическая блокировка `FOR UPDATE`); гонка не уводит счётчик за лимит и не ниже 0 | тест конкуренции (S9), см. Часть VII |
| NFR-11 | Безопасность | все защищённые вызовы — только с валидным `Bearer`-JWT; пароли персонала — pbkdf2_sha256; секреты — из ENV, не в коде | ревью + тесты авторизации |
| NFR-12 | Совместимость | актуальные мобильные браузеры (iOS Safari, Android Chrome) + desktop Chrome/Edge/Safari; мин. ширина экрана **320 px** | кросс-браузерная проверка |

## I.6 Безопасность и приватность (PDPL UAE)

> **Для заказчика.** Продукт собирает минимум личных данных, нужных только для выдачи заказа: номер
> телефона, номер машины, имя и эмират. По закону ОАЭ о персональных данных (PDPL) такие данные нужно
> хранить ограниченное время, защищать при передаче и хранении и уметь удалять по запросу. Ниже —
> опись данных и правила обращения с ними.

### Как это работает (простыми словами)

Когда покупатель оформляет заказ, он указывает телефон (чтобы узнавать его заказы и связаться) и номер
машины (чтобы узнать его у окна выдачи). Эти данные видит сотрудник точки только по заказам своей
точки; администратор сети — по всей сети. Данные передаются по защищённому каналу. В демонстрациях и
скриншотах используются только **выдуманные** телефон и номер машины — настоящие данные людей туда не
попадают.

### Технические детали (для разработки)

**PII-инвентарь (минимальный обязательный набор):**

| Поле | Класс | Где хранится | Правовое основание | Срок хранения (дефолт 📝) | Путь удаления |
|---|---|---|---|---|---|
| `phone` | прямой идентификатор | `User.phone`, `Order.phone` | исполнение заказа (контракт) | привязка к жизни аккаунта; заказы — см. ниже | удаление/анонимизация аккаунта |
| `car_plate` | косвенный идентификатор (выдача) | `User.car_plate`, `Order.car_plate` (UPPERCASE) | исполнение выдачи | срок хранения заказа | очистка поля при анонимизации заказа |
| `name` | имя | `User.name`, `Order.customer_name` | контакт при выдаче | жизнь аккаунта | удаление аккаунта |
| `emirate` | гео-атрибут | `User.emirate`, `Order.emirate` | контекст выдачи/аналитика | жизнь аккаунта/заказа | удаление аккаунта |

📝 **Решение с дефолтом (вынести в BR, Часть IX):**
- **Срок хранения заказов** — по умолчанию **24 месяца** (фискальная отчётность ОАЭ), затем
  анонимизация PII в заказе (телефон/номер машины/имя → обезличить, сумма/состав — оставить для
  аналитики). ❓ подтвердить с заказчиком.
- **`car_plate` как PII** — трактуем как косвенный идентификатор; хранение и удаление — по правилам
  заказа. ❓ подтвердить классификацию.

**Меры (требования):**
- **Транзит** — HTTPS/TLS на проде (обязательно); WS — `wss://`.
- **Покой** — секреты (`jwt_secret`, ключи Stripe/S3) только из ENV; БД-бэкапы — в защищённом
  хранилище.
- **Контроль доступа** — серверный ролевой скоуп: менеджер видит PII только по заказам **своей** точки
  (`FOREIGN_LOCATION → 403`, см. Часть VI); super-admin — по всей сети; клиент — только свои заказы.
- **Маскирование** — в логах и нефискальных выгрузках телефон/номер машины маскировать (например,
  `+9715****0000`, `DEMO ••01`); полный вид — только в защищённых админ-экранах по праву доступа.
  📝 централизованная маскировка PII в логах — задача реализации (в коде прототипа не зафиксирована).
- **Синтетические PII в демо** — обязательное правило: в сидах и скриншотах только выдуманные данные
  (`+971500000000`, `DEMO 0001`).

## I.10 Единицы, деньги и налог (AED, VAT 5 %)

> **Для заказчика.** Все цены — в дирхамах ОАЭ (**AED**). В ОАЭ действует НДС (**VAT**) **5 %**.
> **Важно и требует вашего решения:** в собранном прототипе налоговая строка в чеке **не считается** —
> итог равен сумме напитков минус скидка по купону. Это значит, что VAT нужно либо **«зашить» в цену
> напитка** (модель «включён в цену» — по умолчанию для ОАЭ-розницы), либо **начислять сверху** —
> и это надо настроить до приёма реальных денег.

### Как это работает (простыми словами)

Покупатель видит цену каждого напитка и итог к оплате. Итог считается просто: сумма всех напитков
минус скидка по купону (если применён). Отдельной строки «VAT 5 %» в текущем расчёте нет. Для
розницы в ОАЭ стандартная модель — **цена уже включает VAT** (тогда видимый итог не меняется, налог
выделяется в фискальном чеке «обратным счётом»). Если же выбрать модель «налог сверху», итог к оплате
увеличится на 5 % — это другое поведение, и его нужно сознательно включить.

### Технические детали (для разработки)

**Канон расчёта (`services/order_flow.py`):**
```
order.subtotal = round(Σ(unit_price × quantity), 2)
order.coupon_discount = round(цена_выбранного_напитка, 2)   # купон списывает 1 напиток, иначе 0
order.total = round(order.subtotal − order.coupon_discount, 2)
```
⚠️ **VAT в коде НЕ начисляется** — отдельной налоговой строки нет, `total = subtotal − coupon`. Это
зафиксированное расхождение «код ↔ требование VAT 5 %» (см. спину §«Известные дефекты», п.3, и Часть IX).

**Формат на фронте:** цена напитка — `DHS. X.XX`; итог — `AED {total}`.

#### Пример расчёта чека (реальные числа из сида `seed_grabzi`)

> Цены берутся из каталога-сида (`base_price`): Classic Ice V'60 = **28.00**, Cherry Bomb Ice V'60 =
> **36.00**, Choco Berry Ice V'60 = **34.00**, Melon Spark Ice V'60 = **37.00** AED.

Заказ: **2× Classic Ice V'60 @ 28.00** + **1× Cherry Bomb Ice V'60 @ 36.00**.

**Так, как считает код сейчас (без VAT-строки):**

| Позиция | Кол-во | Цена | Сумма |
|---|---|---|---|
| Classic Ice V'60 | 2 | 28.00 | 56.00 |
| Cherry Bomb Ice V'60 | 1 | 36.00 | 36.00 |
| **Подытог (subtotal)** | | | **92.00 AED** |
| Скидка по купону | | | 0.00 |
| **Итого к оплате (total)** | | | **92.00 AED** |

(Это совпадает с состоянием скриншота `order--loaded`: «2× Classic, 1× Cherry, AED 92».)

**Модель A — «VAT включён в цену» (рекомендуемый дефолт для ОАЭ-розницы) ⚠️ требует настройки:**
итог к оплате **не меняется** (92.00 AED), но в фискальном чеке VAT выделяется обратным счётом:
```
VAT (5 %, включён) = total − total / 1.05 = 92.00 − 87.62 = 4.38 AED
Нетто (без VAT)    = 87.62 AED
К оплате           = 92.00 AED
```

**Модель B — «VAT начисляется сверху» (альтернатива) ⚠️:**
итог к оплате **увеличивается**:
```
Подытог (нетто) = 92.00 AED
VAT 5 %         = 4.60 AED
К оплате        = 96.60 AED
```

> ⚠️ **Решение заказчика (вынести в BR, Часть IX):** выбрать модель VAT (по умолчанию — **A,
> «включён в цену»**), указать **TRN** для фискального чека и при необходимости включить отдельную
> налоговую строку. До этого приём **реальных** денег невозможен (см. §I.3, Stripe-mock). Макет чека
> повторяется в Части IV (экран O1) и в процессе оплаты (Часть VII).
>
> 📝 Дополнительные открытые вопросы по деньгам (вынести в Часть IX): **минимальная сумма заказа** (в
> коде не задана), **номинал и срок купона** за 👎 (в коде жёстко не зафиксированы).

**Прочие единицы и форматы:**
- **Телефон** — `+971…`; канон-валидация бэка `^\+\d{9,15}$` (⚠️ фронт мягче, см. §I.2).
- **Номер машины** (`car_plate`) — обязателен для выдачи, приводится к **UPPERCASE**; PII (§I.6).
- **Эмират** (`emirate`) — поле заказа/профиля.
- **Время** — TZ каждой точки (`Asia/Dubai`); «бизнес-день» — в TZ точки (дневной лимит, см. Часть VII).
- **Идентификаторы** (поля, enum, коды ошибок, маршруты) — латиницей, как в коде.
