# Juicy (drinkit.mediann.dev) — отчёт полного прогона и починки

> Полное исчерпывающее тестирование (каждый эндпоинт + каждое поле; каждый экран + кнопки + полный путь покупки)
> с починкой найденного. Прогон против реального окружения и локального стека на проде-идентичном коде.
> Дата: 2026-06-12.

## Вердикт
- **Бэкенд:** 52 эндпоинта покрыты, **561 автотест зелёный**; найдено **15 багов**, **10 починено** (включая 2 high), 5 low — отложены с обоснованием.
- **Фронтенд:** **55 e2e зелёных** (все экраны клиента и админки, отсутствие мёртвых кнопок, полный путь покупки end-to-end на реальных данных). 4 найденные проблемы (вкл. security) — **починены**.
- Деплой-код бэкенда **байт-в-байт = протестированному** (MD5 `be9d5119…`), поэтому результаты применимы к проду.

| **Итог** | **55 passed** | |

### Найденные проблемы и починка
| # | Severity | Где | Проблема | Статус |
|---|---|---|---|---|
| S1 | 🔴 security | `admin/login` | публично печатались рабочие доступы | **fixed** (показ только при `NODE_ENV!=='production'`) |
| A1 | 🟠 major | `profile` | счётчик «Заказов» из локального стора (всегда 0) | **fixed** (берётся из `GET /api/orders`) |
| A4 | 🟡 minor | `profile` | мёртвые ссылки «Условия/Политика/Связаться» | **fixed** (убраны; вернуть рабочими при наличии реальных URL/контакта) |
| A5 | 🟡 minor | `auth/otp` | автозаполнение OTP из sessionStorage | **fixed** (гард `NODE_ENV!=='production'`) |

Прочее: раздел **«Точки» в админке — мок** (`lib/admin-mock.ts`), и клиентский выбор точки из мока (`lib/data.ts`) — это вне сметы и совпадает с зоной бонус-фичи «часы приёма», которой нужен бэкенд. Оставлено как есть (не в scope).

### Починка тестовой инфраструктуры (дрейф данных)
- `e2e/_helpers.ts`: слаги заменены со старого кофейного мока (`matcha`, `croissant`…) на реальные сок-слаги (`orange-fresh`…).
- `golden-path.spec.ts` и `buttons.spec.ts` переписаны под реальную структуру (нет размеров «мл», категории добавок — кнопки + popover `.animate-popoverUp`, оплата «Оплатить через Stripe»).
- Добавлен `e2e/admin.spec.ts` (12 тестов админки).
- Замечен race: админ-гард может кинуть на login, если `juicy-staff-token` не успел записаться — в тестах укреплён `login()`; на проде для живых пользователей не воспроизводится.

## 3. Окружение
- Прод `drinkit.mediann.dev` = контейнеры `juicy_frontend_prod` + `juicy_api_prod`; репо `/opt/drinkit` (git `main`).
- ⚠️ Локально ранее запущенный `:8000/:3000` — это GRABZI (другой проект), не Juicy.
- Локальный Juicy-фронт грузит данные только в **prod-режиме** (`next build && next start`); в dev (Turbopack) фетч не срабатывал — артефакт окружения, не код (на проде всё грузится).

## 4. Как воспроизвести
```bash
# backend (изолированная БД, тот же код, что на проде)
cd backend && .venv/bin/python -m pytest -q          # 561 passed, 2 xfailed
# локальный стек для e2e
cd backend && DATABASE_URL="sqlite:///./juicy_smoke.db" AUTH_OTP_ENABLED=false .venv/bin/uvicorn app.main:app --port 8011
cd app && echo 'NEXT_PUBLIC_API_URL=http://127.0.0.1:8011' > .env.local && npx next build && npx next start -p 3300
cd app && npx playwright test                         # 55 passed
```

## 5. Что осталось (рекомендации)
1. **WS-авторизация** (B13–B15): добавить токен в WebSocket-подключение и проверку владельца заказа/роли — закрывает утечку статусов чужих заказов. Требует согласованной правки `lib/api.ts` (WS-клиент).
2. Refund «только note» (B11) и FK-проверка addon `categoryId/unitId` (B12) — мелкие валидации.
3. Раздел «Точки»: сделать бэкенд (это и есть бонус «часы приёма») либо скрыть мок-экраны.
4. Вернуть help/legal-ссылки (A4) рабочими, когда будут реальные URL и контакт.

## 6. Итог
Ядро Juicy (каталог, конструктор, заказ, оплата-mock, статусы, realtime, RBAC, i18n) — **протестировано исчерпывающе и работает**; критичные баги (отрицательное количество, дабл-букинг купона, цена бесплатной добавки, валидации PATCH) и security-проблема логина — **починены**. Изменения на ветке `juicy`, бэкенд-код идентичен проду — готово к ревью и деплою.
