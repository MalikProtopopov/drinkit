# Juicy (jooz.ae) — Единое техническое задание

**Продукт:** mobile-web сервис предзаказа для juice bar в ОАЭ (модель Drinkit, curbside-выдача по номеру машины, конструктор напитков с КБЖУ).
**Версия документа:** 1.0 · **Дата:** 2026-06-04 · **Статус:** собрано из источников S1–S11, готово к оценке.
**Назначение:** единое описание функционала, по которому можно (1) разработать продукт и (2) собрать поблочную смету (см. `Juicy_СМЕТА.md`).

---

## Легенда и конвенции

**Пометки статуса:**
- ✅ — реализовано в прототипе `app/` и покрыто e2e;
- 🔧 — поведение только в dev/mock (демо-данные, авто-таймеры, имитация оплаты), в проде не действует;
- 🚧 — не реализовано / только в спеке `BACKEND.md` / нет экрана;
- ⚠️ — изменение «прототип → целевой продукт» или противоречие источников;
- 🧩 — обоснованное допущение в рамках концепции (требует подтверждения заказчиком).

**Единицы и форматы:**
- Деньги — **AED**, в API минорные единицы/decimal-string `"12.50"`; **VAT 5 %** (на старте чеки/фискализация не требуются — ⚠️ см. §0.1, печать чека вне MVP).
- КБЖУ — ккал / белки(г) / жиры(г) / углеводы(г); live-пересчёт `база + Σ(добавки)`.
- Объёмы/дозы рецептов — `ml`/`g` (см. §IV).
- **Номер машины (plate)** — Latin-only (`[A-Z0-9 ]`), потенциальный PII (PDPL UAE).
- Телефон — мультикод стран Залива (UAE `+971`, Kuwait и соседние); ⚠️ в прототипе только `+971`.
- Языки — **EN + AR (RTL)** (целевой продукт); ⚠️ в прототипе UI на RU.
- Идентификаторы (поля, enum, маршруты, статусы, коды ошибок) — латиницей, как в коде.

**Формат функциональных требований:** `Назначение → Что видит пользователь → Данные/API → Действия → Состояния → Бизнес-правила → Оценочный объём`.

**Четыре проекции продукта:** Клиент (mobile-web) · Админка (desktop) · KDS (планшет) · Бэкенд. Связаны сквозными процессами (§VII).

# ЧАСТЬ I. Архитектура и сквозные конвенции

## I.1 Технологический стек

- **Frontend (клиент + админка):** Next.js **16** (App Router) + React **19.2** + TypeScript + Tailwind v4; состояние — Zustand (в прототипе persist в localStorage; в проде — серверное API). ⚠️ Расхождение источников: compass S1 — Next.js 15 / PG 16; КП S4 и код S3 — Next.js 16 / React 19. **Канон — код (Next.js 16).**
- **Backend (план, S2):** FastAPI (Python 3.12) + PostgreSQL + Redis (rate-limit/кэш/pub-sub) + S3-совместимое хранилище медиа. 🚧 в прототипе бэкенда нет — `lib/data.ts` (каталог) и Zustand (корзина/заказы), всё mock.
- **Платежи:** ⚠️ **Stripe** (целевой; карта, оплата сразу). В спеке S2 фигурирует Tap Payments — заменяется на Stripe.
- **Реалтайм:** SSE для статусов клиенту; WebSocket для KDS (🚧).
- **Хостинг:** регион ОАЭ (`me-central-1`) либо EU; домен **jooz.ae**; PDPL-комплаенс.
- **Вход:** по QR-коду на здании → mobile-web.

## I.2 Аутентификация и роли

- **Клиент (целевой):** ⚠️ **без OTP** на старте. Лёгкая регистрация: имя + телефон (мультикод) + номер машины. Пользователь создаётся в БД, заказ привязывается к **номеру телефона** (номера машин меняются: один телефон — несколько машин; встречается «police car» вместо номера). Полноценный личный кабинет — позже. Задел: возможность срастить со старой базой клиентов (лояльность). 🚧 OTP/SMS — на этап мобильного приложения.
- **Персонал (S2 §0):** JWT с `roles: string[]` и `outletScope: string[] | "*"`. Роли: `super_admin`, `catalog_admin`, `outlet_manager`, `barista` (KDS), `finance`, `support`. ⚠️ Для MVP соков критичны `super_admin`, `catalog_admin`, `outlet_manager`, `barista`; `finance`/`support` — задел.

## I.3 Деньги, идемпотентность, заголовки
- VAT 5 % считается на сервере; `subtotal → vat → total`. В прототипе `computeCartTotal` (`VAT_RATE=0.05`). На старте чек/TRN не печатаются (⚠️ C6).
- Идемпотентность: `Idempotency-Key` обязателен для мутаций (создание заказа и т. п.).
- Заголовки запроса (S2 §1): `Authorization: Bearer <jwt>`, `Accept-Language: en|ar`, `X-Outlet-Id: <outletId>`, `Idempotency-Key: <uuid>`.

## I.4 Формат ошибок (RFC 9457) и пагинация
Единый конверт ошибки + коды — см. §VIII.3. Пагинация списков — курсорная (`?cursor=…`).

## I.5 Локализация (целевой EN+AR)
- ⚠️ Целевой продукт: EN + AR с RTL **с самого старта** (аудитория — местные, часто без английского). Прототип — RU.
- Разделение (S7 E3): UI-строки — на стороне разработки (часть генерируется/хардкодится, можно прогнать через ИИ); контент (названия/описания напитков, ингредиенты) — заполняется заказчиком через админку на EN/AR.
- RTL-готовность: logical properties (`margin-inline-*`, `ms-/me-`), зеркалирование направленных иконок; НЕ зеркалить логотипы, фото товаров, поля цифр (`dir="ltr"` для телефона/номера/карты).
- Шрифты (S1): латиница — Inter/Manrope; арабский — IBM Plex Sans Arabic / Tajawal.

## I.6 Дизайн-система (S1, прототип)
- Первичный цвет индиго **#4A56E2** (-600 `#3B47C9`, -100 `#E5E7FC`); бежево-сиреневые подложки продуктов (`#F5EFE7`, `#E8DCD0`, `#EFE6F0`, `#D8C8E0`); semantic success `#16A34A` / warning `#F59E0B` / error `#DC2626`.
- Радиусы pill `9999`, карточки 16–24; spacing 4/8/12/16/24/32; тач-таргеты ≥ 44×44.
- mobile-frame: 100 % ширины на телефоне, центрирование в ~390 px на десктопе; скроллятся внутренние контейнеры; safe-area через `env(safe-area-inset-*)`.
- Медиа: каждому напитку — короткая MP4-петля (`<DrinkMedia>`), фолбэк — SVG `<DrinkArt>`.

# ЧАСТЬ II. Модель данных

Канон — `BACKEND.md` (S2 §2, ~27 таблиц), скорректировано под целевой продукт (Stripe, без OTP, EN/AR). Ниже — сводка ключевых сущностей; полные DB-поля — в S2.

## II.1 Каталог
- **Outlet** — `id`, `name`, `address`, `city`, `geo{lat,lng}`, `hours` (WeeklyHours), `isOpen` (computed, TZ Asia/Dubai), `closingSoon?`, `trn?`, `printerWebhook?` (V2). В прототипе 4 точки (Bay Avenue, Emaar Square, Dubai Marina, City Centre Mirdif).
- **Category** — `slug`, `name{en,ar(,ru)}`, `emoji?`, `sortOrder`, `visible`. ⚠️ Целевые разделы: Classic / Signature Mixes / Смузи-Авокадо / Wellness-Nutrition (тест-данные — текущие 8 категорий).
- **Product** — `id`, `slug`, `categorySlug`, `name{en,ar}`, `description{en,ar}`, `badge?` (`BESTSELLER|NEW|LIMITED|SEASONAL`), `baseNutrition` (ккал/Б/Ж/У/сахар/клетчатка/вес, для размера M), `variants[]`, `addonGroupBindings[]`, `media{videoUrl?,posterUrl(req),pngUrl?,bgColor}`, `tags[]`, `inStock` (per-outlet), `visible`, `prepTimeSec`, `sortOrder`.
- **Variant** — `code` (S/M/L), `name`, `volumeMl?`, `weightG?`, `priceAed`, `nutritionMod` (дельта к базе), `inStock`.

## II.2 Конструктор (детально — §IV)
- **Addon** — `id`, `name{en,ar}`, `pngUrl` (req), `unit` (`shot|scoop|piece|portion|ml|g`), `doseAmount`+`doseUnit` (`g|ml`), `nutritionPer100`, `pricePerUnitAed`, `allergens[]`, `tags[]`, `visible`. Формула вклада КБЖУ: `(doseAmount × qty / 100) × nutritionPer100`.
- **AddonGroup** — `slug`, `name`, `selectionType` (`single|multi|counter`), `items[]` (addon ids), `visible`.
- **AddonGroupBinding** (product↔group, тут групповые правила) — `groupSlug`, `required`, `minTotal?`, `maxTotal?`, `defaultAddonIds?`, `overrides[]`.
- **AddonOverride** (product↔addon) — `minUnits?`, `maxUnits?`, `defaultUnits?`, `pricePerUnitAedOverride?`, `free?`, `doseAmountOverride?`, `hidden?`.
- **CompatibilityRule** — кросс-аддонные правила (forbid / auto-add / require / discount-percent). См. §IV.3.

## II.3 Заказ
- **Order** — `id`, `number` (публичный 4-значный), `createdAt`, `outletId`, `items[]`, `subtotalAed`, `vatAed`, `discountAed`, `totalAed`, `status` (OrderStatus), `statusHistory[]`, `payment{method, providerChargeId?, providerStatus?}`, `pickup{car{emirate,plate}, scheduledFor?, arrivedAt?}`, `customer{id?,name?,phone}`, `baristaId?`, `prepEtaSec?`.
- **OrderItem** — `productId`, `variantCode`, `quantity`, `customName?`, `addons[{addonId,groupSlug,units}]`, + снапшоты `productNameSnapshot`, `unitPriceAedSnapshot`, `nutritionSnapshot` (изменение каталога не ломает историю).
- **OrderStatus**: `CREATED → PAID → ACCEPTED → PREPARING → READY → PICKED_UP` (+ `CANCELLED`, `REFUNDED` 🚧). В прототипе клиентский тип статуса без CANCELLED/REFUNDED.

## II.4 Пользователи и персонал
- **User** — `id`, `phone`, `name?`, `defaultEmirate?`, `defaultCarPlate?`, `preferredLocale` (`en|ar`), `marketingOptIn`, `blocked`.
- **StaffUser** — `id`, `email`, `name`, `roles[]`, `outletScope[]|"*"`, `twoFactorEnabled` (V2), `disabled`.

# ЧАСТЬ IV. Конструктор и формулы напитков (ключевой раздел)

> Канон требований — уточнения Малика (S11) + созвон (S7); канон модели — `BACKEND.md` §2.4–2.6; «как есть» — `lib/data.ts` (11 addon-групп) и карточка `product/[slug]`. Каталог — **тест-данные** (🧩, финальный файл от Абдулазиза заменит данные без изменения движка).

## IV.1 Принцип (S11): гибкость и масштабируемость
Формула напитка и правила задаются **данными**, а движок один. Новые комбинации/ингредиенты/правила добавляются в админке без изменения кода. Три требования заказчика:
1. **Состав рецептов** — ингредиенты + граммовки (формат формулы);
2. **Правила сочетания** — что с чем нельзя смешивать;
3. **Стоп-логика/лимиты** — когда сказать пользователю «стоп».

## IV.2 Модель данных конструктора (4 слоя + правила)

**Слой A — Addon (глобальное определение ингредиента/добавки).**
`unit` (`shot|scoop|piece|portion|ml|g`), `doseAmount`+`doseUnit` (`g|ml`), `nutritionPer100` (КБЖУ на 100 г/мл), `pricePerUnitAed`, `allergens[]`. Это и есть «ингредиент с граммовкой» (S11.1). Вклад в КБЖУ: `(doseAmount × qty / 100) × nutritionPer100`.

**Слой B — AddonGroup (способ выбора).**
`selectionType`: `single` (ровно один, radio) / `multi` (несколько, чекбоксы) / `counter` (счётчик units на аддон). В прототипе 11 групп: supplements(single), foam(counter≤2), cream(single), espresso(counter≤3), tapioca(counter≤2), toppings(multi), sugar(single), syrups(multi), citrus(counter≤2), protein(single), collagen(single).

**Слой C — AddonGroupBinding (product↔group): ГРУППОВЫЕ ОГРАНИЧЕНИЯ.**
`required` (надо ли выбрать), `minTotal?`/`maxTotal?` (лимит суммарных units по группе — это и есть **стоп-логика по группе**, S11.3), `defaultAddonIds?`, `overrides[]`.

**Слой D — AddonOverride (product↔addon): per-добавка в контексте блюда.**
`minUnits?`/`maxUnits?` (напр. ≤4 шотов — стоп по добавке), `defaultUnits?`, `pricePerUnitAedOverride?`/`free?`, `doseAmountOverride?`, `hidden?`. Резолв: `effPrice = override ?? (free?"0":base)`, `effDose = doseOverride ?? base`, `effMax = maxUnits ?? ∞`.

## IV.3 Правила сочетания — CompatibilityRule (S11.2)
Отдельная сущность, редактируется в админке без кода:
```ts
type CompatibilityRule = {
  id: string; scope: "product"|"global"; productSlug?: string;
  when: { addonId: string; minUnits?: number }[];   // конъюнкция: все условия выбраны
  effect:
    | { type: "forbid";  addonId: string; message?: {en,ar} }   // запрет несовместимого → 409 ADDONS_INCOMPATIBLE
    | { type: "auto-add"; addonId: string; units: number }      // авто-добавление
    | { type: "require";  groupSlug: string }                   // обязать выбор в группе
    | { type: "discount-percent"; percent: number };            // комбо-скидка
};
```
Примеры: `[milk-oat] → forbid milk-regular` (нет двойного молока); `[shot-decaf] → forbid shot-regular` (кофеин/декаф взаимоисключение); `[croissant, hot-chocolate] → discount-percent 10` (комбо).

## IV.4 Движок («сердце формул»)
При каждом изменении выбора клиент шлёт `POST /products/{slug}/preview {variantCode, addons[]}`. Сервер прогоняет AddonGroupBinding + CompatibilityRules и возвращает `{ allowedAddons, forbidden[], autoAdded[], priceAed, nutrition }`. Клиенту не нужно знать формулы. Менеджер тестирует правила кнопкой «Проверить» (`POST /admin/dryrun-preview`).

## IV.5 Стоп-логика и поведение UI (S11.3)
- Лимиты: `maxTotal` по группе, `maxUnits` по добавке, `forbid` по правилу, `inStock=false` на любом уровне.
- При достижении лимита: кнопка «+» блокируется, показывается подсказка-причина; запрещённые добавки — задизейблены с локализованным сообщением из правила.
- Доп. лимиты сборки (🧩, на согласование): максимум добавок на напиток / макс. объём — параметризуемо.

## IV.6 Привязка к разделам (целевой каталог)
- **Classic** — минимальная кастомизация (lightAddons-аналог).
- **Signature Mixes** — готовые формулы с ограниченной правкой.
- **Смузи/Авокадо** — слоистая сборка (база → мусс → топпинги).
- **Wellness/Nutrition** — по цели, кастомизация минимальна.

## IV.7 Снапшоты
Цены/КБЖУ/дозы снапшотятся в позицию заказа (`order_item_addons`), изменение каталога не ломает историю.

# ЧАСТЬ VII. Сквозные процессы

1. **Регистрация без OTP** ⚠️ — телефон+имя+plate → создание `User` в БД → заказ привязан к телефону. Задел: срастка со старой базой лояльности (миграция, §C11).
2. **Оплата Stripe** ⚠️ — `POST /payments/intent` → оплата (карта; Apple/Google Pay опц.) → webhook `payment_intent.succeeded` → заказ `PAID`. Без чеков/возвратов на старте.
3. **Жизненный цикл заказа + curbside** — `CREATED→PAID→ACCEPTED→PREPARING→READY→PICKED_UP`; клиент жмёт `Я приехал` (`/orders/{id}/arrived`) → KDS показывает номер машины → бариста выносит. В прототипе статусы авто-симулируются 🔧; в проде — SSE-канал.
4. **Конструктор/формулы** — см. §IV (preview-движок: binding + compatibility rules → allowed/forbidden/auto-add/price/nutrition).
5. **KDS realtime** 🚧 — `WS /kds/stream`: `order_new`, `customer_arrived{orderId, carPlate}`; статусные действия бариста.
6. **Лояльность-задел** — структура под «каждый N-й бесплатный» и перенос истории (реализация — V2).
7. **Фишки Drinkit (опция, C12)** — (а) ваучер при превышении времени ожидания (лайк/дизлайк → ваучер); (б) «переделать напиток» = заявка особого типа: один напиток, без возврата/доплаты, оповещение менеджеру.

# ЧАСТЬ VIII. Справочники

## VIII.1 Статусы заказа
`CREATED` (создан, ожидает оплаты) · `PAID` (оплачен) · `ACCEPTED` (принят бариста) · `PREPARING` (готовится) · `READY` (готов) · `PICKED_UP` (выдан) · `CANCELLED`/`REFUNDED` 🚧 (вне MVP).

## VIII.2 Константы
VAT 5 % · слоты выдачи шаг 15 мин от now+15 · кастомное имя ≤30 · plate `^[A-Z0-9 ]{1,12}$` · телефон UAE `^\+971(50|52|54|55|56|58)\d{7}$` (⚠️ расширить на коды Залива) · номер заказа от 1247 (прототип) · лимиты добавок: foam ≤2, espresso ≤3, tapioca ≤2, citrus ≤2 (тест-данные).

## VIII.3 Коды ошибок (S2 §9)
`VALIDATION_ERROR`(400) · `AUTH_REQUIRED`(401) · `FORBIDDEN`(403) · `NOT_FOUND`(404) · `OUTLET_CLOSED`(409) · `PRODUCT_OUT_OF_STOCK`(409) · `ADDON_OUT_OF_STOCK`(409) · `ADDON_BINDING_VIOLATED`(409) · `ADDONS_INCOMPATIBLE`(409) · `ORDER_NOT_CANCELLABLE`(409) · `PAYMENT_FAILED`(422) · `RATE_LIMITED`(429) · `INTERNAL`(500). 🚧 OTP-коды (`OTP_INVALID`, `OTP_THROTTLED`) — на этапе приложения.

## VIII.4 Внешние интеграции
- ⚠️ **Stripe** (карта; Apple/Google Pay опц.; вебхук с подписью).
- 🚧 SMS-провайдер (Twilio/Unifonic) — только при возврате OTP/уведомлений (этап приложения).
- 🚧 Учётная/POS-система (аналог iiko/Shopify ERP) — выберет Абдулазиз; на старте не интегрируем, заложить возможность.
- Аналитика-события `/events`; Sentry/OpenTelemetry (NFR).

## VIII.5 Глоссарий
блюдо = продукт = напиток (`Product`); точка = `Outlet`; добавки = доп = add-on; бариста = KDS-оператор; конструктор = кастомайзер.

# ЧАСТЬ IX. Качество, ограничения, открытые вопросы

## IX.1 NFR (целевой, с учётом S7/S9)
- **Нагрузка низкая:** ~до 100 заказов/день на точку, тестовый режим, трафик специально не нагоняют (S9 F3). p95 `/catalog` < 200 мс, `POST /orders` < 400 мс (S2 — ориентир).
