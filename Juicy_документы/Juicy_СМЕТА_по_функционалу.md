# Juicy — смета разработки по функционалу

> Смета построена по функциональным требованиям (32 user stories, документ «Функциональные
> требования: User Stories по ролям и платформам»). Каждая строка сметы — блок функционала с
> привязкой к историям, оценкой по **бэкенду / фронтенду / дизайну** и стоимостью в трёх тарифах.
> Убрали блок — сразу видно, какие пользовательские сценарии исчезают.
>
> **Дата:** 05.06.2026 · Валюта: ₽, цены без НДС · Оценка действительна 30 дней

**Как выбрать.** Если цель ближайшего месяца — проверить, будут ли заказы, берите **Тариф 1 (ядро)**

экономит деньги на горизонте 6+ месяцев (меньше переделок). **Тариф 3** показывает честную стоимость
полной production-реализации командой — выбирайте его, если сразу нужны масштаб, качество и передача
поддержки внутренней команде.

## 3. Что вы получите на выходе (каждый тариф)

1. **Работающий публичный сайт** (mobile-web) на вашем домене: каталог → конструктор → корзина → оплата → выдача по статусам.
2. **Админ-панель** с двумя ролями: супер-админ (каталог, менеджеры) и менеджер заказов.
3. **Приём реальных оплат** через Stripe (после подключения вашего аккаунта).
4. **Исходный код** в вашем репозитории — без vendor-lock на исполнителя.
5. Документация и поддержка — по уровню тарифа (см. §1).

## 7. Условия и допущения

1. **Ставки:** Тариф 1 — 2 000 ₽/ч · Тариф 2 — 2 900 ₽/ч · Тариф 3 — 3 500 ₽/ч (средневзвешенные по команде). Цены без НДС; валюта и ставки корректируются при согласовании договора.
2. **Коэффициенты трудоёмкости** между тарифами: ×1,6 (Тариф 2) и ×2,3 (Тариф 3) к базовым часам — за счёт тестов, ревью, архитектуры, QA и документации. Исключение — блок 13 «Дизайн» (см. примечание §2.1).
3. Оценка предполагает **переиспользование готового прототипа** клиентского приложения и админки. Отказ от прототипа (дизайн с нуля) увеличивает фронтенд/дизайн-часы в ~2–2,5 раза.
4. До старта необходимо закрыть открытые вопросы требований (минимум: провайдер SMS, одна точка или сеть, валюта/налог в чеке) — влияют на модель данных.
5. **Этапы оплаты (предложение):** 50 % предоплата · 50 % по приёмке. Для Тарифа 3 — 40 % / 30 % (демо середины срока) / 30 %.
6. Приёмка — по критериям AC соответствующих user stories.

## 8. Частые вопросы

**Можно начать с Тарифа 1, а потом «повысить» до Тарифа 2/3?**
Да, это рабочая стратегия «сначала проверить спрос». Честно: часть кода гипотезы при этом
перерабатывается — повторное использование обычно 50–70 %, поэтому суммарно «Тариф 1 → апгрейд»
выйдет на ~20–30 % дороже, чем сразу Тариф 2. Это плата за снятие риска: если гипотеза не взлетит,
вы потеряете 320 тыс, а не 760.

**Можно добавить опции позже, после запуска ядра?**
Да, любой модуль O1–O6 подключается отдельным мини-этапом по ценам этой сметы (в течение срока её
действия). Архитектура ядра проектируется так, чтобы опции вставали без переделок.

**Что если в ходе работы появятся новые требования?**
Новые user stories оцениваются по той же модели (часы Б/Ф/Д × ставка тарифа) и согласуются до
реализации — смета остаётся прозрачной.

**Почему Тариф 3 дороже, если функции те же?**
Вы платите не за «те же экраны», а за то, чего не видно на скриншотах: тесты, которые не дают
сломать оплату при доработках; мониторинг, который будит команду раньше, чем клиенты заметят сбой;
документацию, которая позволит любой команде продолжить разработку без нас.

## 9. Как читать смету вместе с требованиями

Каждая строка сметы ссылается на ID user stories (PUB-…, ADM-…) из документа функциональных
требований — там по каждой истории расписаны критерии приёмки и приложены скриншоты прототипа.
Связка «история → блок сметы» позволяет: убрать блок — и сразу видно, какие сценарии исчезают;
добавить требование — и понятно, какой блок дорожает.
