## F19 — UX-аудит админки (после F17)
Цель — чтобы менеджеру и супер-админу было максимально удобно работать каждый день, а не просто «функции есть».

**Шаг 1. Аудит каждого экрана** — пройти все экраны админки (вход, заказы, деталка заказа, каталог: напитки/категории/добавки/единицы/связки, менеджеры, клиенты, платежи, купоны, дашборд, возвраты) и для каждого зафиксировать в `docs/ADMIN_UX_AUDIT.md`:
- **Цель экрана** — какую задачу пользователь приходит решать и как часто (например: заказы — «быстро увидеть новые и взять в работу», много раз в день; единицы измерения — раз в месяц).
- **Главное действие** — доступно ли оно в 1–2 клика, видно ли без скролла.
- **Проблемы** — лишние клики, нет поиска/фильтров там где списки длинные, нет пустых состояний, непонятные ошибки, нет подтверждений на опасные действия, нет индикации загрузки.

**Шаг 2. План доработок** — там же, в виде таблицы: проблема → решение → приоритет (P1 — мешает ежедневной работе, P2 — заметно, P3 — косметика).

**Шаг 3. Реализация** — выполнить все P1 и P2. Ориентиры удобства:
- Самый частый сценарий (новый заказ → взять в работу → провести по статусам) — минимум кликов, realtime-обновление без F5, звуковая/визуальная индикация новых заказов.
- Списки: поиск, фильтры, сортировка, пагинация — везде, где записей может стать > 20.
- Формы: валидация на месте, сохранение без потери введённого, понятные сообщения об ошибках.
- Опасные действия (удаление, возврат) — подтверждение и возможность отмены, где возможно.
- Дашборд отвечает на вопрос «как идут дела сегодня» за 5 секунд без настройки.

**Результат:** `docs/ADMIN_UX_AUDIT.md` (аудит + план + отметки о выполнении) и коммиты `feat(F19): ...` по доработкам.

## Чистота кода и архитектуры (обязательный регулярный рефакторинг)
Цель — поддерживать фундамент, на который удобно ставить новые фичи. Говнокод не накапливаем: чинить сразу дешевле, чем переписывать потом.

**Когда проводить ревизию:** после каждых **3 завершённых фич** (т.е. после F03, F06, F09, F12, F15, F18) — отдельный проход по коду перед началом следующей фичи.

**Чек-лист ревизии:**
1. **Размер файлов** — файл > ~400 строк это сигнал, > 600 — обязательно разбить (router → выделить service; модуль → разрезать по доменам).
2. **Слои не смешаны** — в роутерах нет бизнес-логики и сырых запросов к БД (только валидация + вызов сервиса); в сервисах нет HTTP-деталей; модели/схемы не содержат логики. На фронте: страницы не содержат fetch-логику напрямую — выносить в api-клиент/хуки.
3. **Дубликация** — одинаковый код в 2+ местах → выносить в общий helper/зависимость (расчёт цены, пагинация, проверки прав, формат ответов).
4. **Мёртвый код** — неиспользуемые функции, импорты, закомментированные блоки, устаревшие эндпоинты — удалять, не хранить «на всякий случай».
5. **Длинные функции** — функция > ~50 строк или с вложенностью > 3 уровней → декомпозировать.
6. **Единообразие** — одинаковые задачи решаются одинаково во всём проекте (один стиль ошибок API, один способ авторизации в эндпоинтах, одна схема именования).
7. **Тесты после рефакторинга зелёные** — рефакторинг не меняет поведение; полный прогон тестов обязателен.

**Результат ревизии:** отдельный коммит `refactor: ревизия после FXX — <что сделано>` (не смешивать с фичевыми коммитами). Если всё чисто — зафиксировать это в коммит-сообщении следующей фичи не нужно, просто продолжать.

## Деплой на тест-сервер
Тест-сервер `72.56.72.47` автоматически подтягивает `main` — настроено, делать ничего не нужно:
- Каталог проекта: `/opt/drinkit`
- Cron: каждые 30 минут запускается `/usr/local/bin/drinkit-pull.sh` (`git fetch` + `git reset --hard origin/main`)
- Лог: `/var/log/drinkit-pull.log`

Т.е. достаточно `git push origin main` после фичи — максимум через 30 минут код окажется на сервере.

### Доступ к серверу (SSH-ключ, без пароля)
- Подключение: `ssh drinkit` (алиас в `~/.ssh/config` → root@72.56.72.47, ключ `~/.ssh/id_ed25519`)
- Форс-пулл немедленно, не дожидаясь cron: `ssh drinkit /usr/local/bin/drinkit-pull.sh`
- Посмотреть лог пуллов: `ssh drinkit tail -20 /var/log/drinkit-pull.log`
- Пароли и приватные ключи в репозиторий не коммитятся — доступ только по ключу с этой машины.
